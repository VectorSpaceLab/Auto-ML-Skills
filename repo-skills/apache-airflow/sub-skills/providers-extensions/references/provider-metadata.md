# Provider Metadata

Provider metadata connects an installed provider package to Airflow discovery, generated docs, provider indexes, connection types, operators, sensors, hooks, triggers, extra links, and decorators.

## File Ownership

- `provider.yaml` is the main human-edited metadata source for a provider.
- `get_provider_info.py` is generated from provider metadata. Do not hand-edit it; update metadata and regenerate through the repository’s provider tooling.
- Provider `pyproject.toml` files are largely generated. Dependency and optional-dependency sections are explicitly preserved and may be edited in place when dependencies change; most other changes should go through templates/generation.
- Provider docs indexes, README files, and provider info modules may be generated from `provider.yaml` and templates. Treat generated headers such as `NOTE! THIS FILE IS AUTOMATICALLY GENERATED AND WILL BE OVERWRITTEN!` as authoritative.
- Provider release classification and bulk documentation preparation are release-manager workflows. This sub-skill can identify metadata edits and generated-file cautions, but broad release-manager classification belongs to contribution tooling.

## Required Provider Shape

A provider directory normally contains:

- `provider.yaml`
- `pyproject.toml`
- `src/airflow/providers/<provider_id>/__init__.py`
- `src/airflow/providers/<provider_id>/get_provider_info.py`
- optional `operators/`, `sensors/`, `hooks/`, `triggers/`, `decorators/`, `transfers/`, `utils/`, and `example_dags/`
- provider docs such as `docs/index.rst`, `docs/changelog.rst`, and how-to files
- tests under `tests/unit/...`

The package naming contract is:

- `provider.yaml` key `package-name`: `apache-airflow-providers-<provider-id>`
- `pyproject.toml` project name: same package name
- `pyproject.toml` entry point group: `apache_airflow_provider`
- entry point target: `airflow.providers.<provider>.get_provider_info:get_provider_info`
- `get_provider_info()` return key `package-name`: same package name

## Common `provider.yaml` Keys

- `package-name`: provider distribution name.
- `name`: display name.
- `description`: provider description, often RST-capable multiline text.
- `state`: provider maturity state such as `ready`.
- `lifecycle`: lifecycle status such as `production`.
- `source-date-epoch`: release/generation timestamp used by provider tooling.
- `versions`: release history maintained by release managers; do not bump manually except for coordinated provider version dependency cases directed by maintainers.
- `integrations`: integration names, docs URLs, tags, logos, and how-to guide paths.
- `operators`, `sensors`, `hooks`, `triggers`: lists of Python modules grouped by integration name.
- `connection-types`: hook class, hook display name, connection type, UI behavior, and custom fields.
- `extra-links`: fully qualified `BaseOperatorLink` classes.
- `task-decorators`: decorator class names and decorator names.
- `config`: provider-specific configuration sections and options.

Provider families may also define sections such as `transfers`, `dialects`, `notifiers`, `secrets-backends`, `logging`, `executors`, or other extension registries.

## Metadata Edit Recipes

### Add An Operator Or Sensor

1. Add the implementation module and focused tests.
2. Add the module path under `operators` or `sensors` for the matching `integration-name`.
3. Add or update how-to docs for user-visible behavior.
4. If the operator/sensor uses a new trigger, add the trigger module under `triggers`.
5. Regenerate provider info/docs with repository tooling before finalizing a PR.

### Add A Hook And Connection Type

1. Add the hook class with `conn_name_attr`, `default_conn_name`, `conn_type`, and `hook_name`.
2. Add hook tests covering connection lookup and UI behavior.
3. Add the hook module under `hooks`.
4. Add a `connection-types` entry with `hook-class-name`, `hook-name`, and `connection-type`.
5. If custom fields are exposed, add matching `ui-field-behaviour` and `conn-fields` metadata.

### Add An Extra Link

1. Add a `BaseOperatorLink` subclass.
2. Attach it to an operator or register it globally through plugin/provider metadata.
3. Add its class path under `extra-links`.
4. Test URL construction and runtime value persistence if the link uses XCom.

### Add Dependencies

1. Edit the preserved dependency section in provider `pyproject.toml`.
2. Keep minimum versions compatible with the provider’s supported Airflow and Python versions.
3. If optional functionality needs an extra, add it under `[project.optional-dependencies]`.
4. Run the repository’s dependency update/pre-commit workflow when available; the generated file comments identify the expected command.

### Update Changelog Or Docs

- Provider packages do not use Airflow core newsfragments. User-visible provider changes belong in that provider’s docs changelog.
- Generated docs indexes and README files should be regenerated, not manually edited.
- Avoid changelog entries for test-only, CI-only, or internal refactors unless maintainers request them.

## Generated File Cautions

- Do not hand-edit `get_provider_info.py`; its header states it is generated and overwritten.
- Do not hand-edit generated provider `pyproject.toml` fields outside the dependency sections marked as preserved.
- Do not manually edit generated provider indexes or README content unless maintainers explicitly identify a non-generated section.
- If a PR needs a workaround or version cap rather than a complete fix, follow the repository’s tracking-issue policy and include the tracking issue URL at the workaround site.

## Bundled Checker

This sub-skill includes `scripts/check_provider_metadata.py`, a safe read-only checker. It performs local consistency checks and prints warnings/errors without network access or repository-specific absolute paths.

Run it from the sub-skill directory or with an explicit path:

```bash
python scripts/check_provider_metadata.py --provider-dir <provider-dir>
```

It checks:

- presence of `provider.yaml`, `pyproject.toml`, and `get_provider_info.py`
- package name consistency between `provider.yaml`, `pyproject.toml`, and `get_provider_info()` text
- presence of the `apache_airflow_provider` entry point
- entry point target ending in `get_provider_info:get_provider_info`
- common metadata sections and generated-file warning headers
- existence of Python modules listed in common metadata sections when the source tree is present

The checker is intentionally conservative. Warnings are prompts for human review; errors indicate likely broken metadata alignment.

## What The Checker Does Not Do

- It does not import the provider package, execute provider code, or contact external services.
- It does not regenerate files.
- It does not validate the full provider schema.
- It does not classify release notes or bump versions.
- It does not replace project test suites or provider documentation generation.
