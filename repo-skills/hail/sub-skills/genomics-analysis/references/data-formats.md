# Dense Genomics Data Formats

Use this reference to choose import options, interpret Hail schemas, and avoid format/reference mistakes.

## Native Hail Formats

### MatrixTable `.mt`

A native MatrixTable is the preferred cache and long-term intermediate for dense genotype data. It stores row, column, entry, and global schemas efficiently and reads faster than text or genetics interchange formats.

```python
mt.write("cohort.mt", overwrite=True)
mt = hl.read_matrix_table("cohort.mt")
mt = mt.checkpoint("stage.mt", overwrite=True)
```

Use `.mt` for iterative Hail analysis. Use VCF/PLINK/BGEN exports only when another tool requires those formats.

### Table `.ht`

A native Table is one-dimensional. Use it for sample annotations, phenotype/covariate tables, interval tables, variant annotations, association results, related-pair outputs, and VEP rows. Do not use a Table as a substitute for dense genotype entries unless coordinate-form entries are truly required.

## VCF

### Import Expectations

`hl.import_vcf` accepts one or more files or glob patterns. Files must have compatible headers and the same samples in the same order when imported together.

Recommended options:

```python
mt = hl.import_vcf(
    paths,
    reference_genome="GRCh38",
    header_file=None,
    force_bgz=False,
    array_elements_required=False,
    contig_recoding=None,
    skip_invalid_loci=False,
)
```

Compression guidance:

- `.vcf`: uncompressed; parallelism depends on splittability.
- `.vcf.bgz`: block-gzipped; preferred for large VCFs.
- `.vcf.gz`: often ambiguous. Use `force_bgz=True` only if it is actually block-gzipped. Use `force=True` only for standard gzip and expect slow single-core parsing per file.

### VCF Field Layout in Hail

| VCF component | Hail axis | Typical Hail field |
| --- | --- | --- |
| CHROM/POS | row key | `locus` |
| REF/ALT | row key | `alleles`, with reference first |
| ID | row | `rsid` |
| QUAL | row | `qual` |
| FILTER | row | `filters: set<str>`; PASS means empty set |
| INFO | row | `info: struct` |
| Sample IDs | column key | `s` |
| FORMAT fields | entry | `GT`, `AD`, `DP`, `GQ`, `PL`, etc. |

`GT` and fields named in `call_fields` are parsed as `call`; other FORMAT fields follow header types. INFO or FORMAT arrays are represented as arrays when the VCF header `Number` is not 1.

### VCF Export Expectations

`hl.export_vcf` uses:

- Row fields: `rsid`, `qual`, `filters`, `info`.
- Entry fields: all fields in the entry struct become FORMAT fields.
- Column key: sample IDs must be string-like and suitable for VCF columns.
- Reference: contig header lines come from the row `locus` reference genome.

Warnings:

- Filtering samples or entries does not update imported `info.AC`, `info.AF`, `info.AN`, or similar fields. Recompute with `hl.variant_qc` and overwrite/update `info` before export.
- Split multiallelic rows may leave allele-specific INFO arrays with old lengths. Use `a_index - 1` to select split-allele values when scalar values are needed.
- Use `.vcf.bgz` and optionally `tabix=True`; avoid standard `.gz` output.
- Use `metadata=hl.get_vcf_metadata(original_vcf)` when preserving header descriptions and Number/Type annotations matters.

## PLINK BED/BIM/FAM

`hl.import_plink` reads binary SNP-major PLINK files. Individual-major files must be converted with PLINK before Hail import.

Input files:

- `.bed`: binary genotype matrix.
- `.bim`: variant records with chromosome, rsid, centimorgan position, base-pair position, and alleles.
- `.fam`: family/sample records with family ID, individual ID, parents, sex, and phenotype.

Hail schema:

| Source | Axis | Hail field |
| --- | --- | --- |
| BIM chromosome/position | row key | `locus` |
| BIM alleles | row key | `alleles`; A2 is reference by default when `a2_reference=True` |
| BIM variant ID | row | `rsid` |
| BIM centimorgans | row | `cm_position` |
| FAM individual ID | column key | `s` |
| FAM family/parents | column | `fam_id`, `pat_id`, `mat_id` |
| FAM sex | column | `is_female` |
| FAM phenotype | column | `is_case` or `quant_pheno` |
| BED genotype | entry | `GT: call` |

Important options:

- `quant_pheno=True` imports FAM phenotype as a numeric `quant_pheno`; otherwise case-control `is_case` is used.
- `missing` controls phenotype missing values beyond built-in case-control missing markers.
- `a2_reference=False` flips which BIM allele is treated as reference.
- PLINK chromosome codes get default recoding for common human references; inspect `mt.describe()` and row keys.

## BGEN

`hl.import_bgen` reads BGEN files as dense MatrixTables. Each file must have an `.idx2` index generated by `hl.index_bgen` using the same reference genome/import assumptions.

Workflow:

```python
hl.index_bgen("cohort.bgen", reference_genome="GRCh38")
mt = hl.import_bgen(
    "cohort.bgen",
    entry_fields=["dosage"],
    sample_file="cohort.sample",
)
```

Hail schema:

| BGEN component | Axis | Hail field |
| --- | --- | --- |
| Variant identifying block | row key | `locus`, `alleles` |
| Variant ID / rsID | row | `varid`, `rsid` when requested |
| Sample IDs | column key | `s` |
| Hard calls | entry | `GT` when requested; missing if probabilities do not have a unique maximum |
| Probabilities | entry | `GP`, array `[P(0/0), P(0/1), P(1/1)]` |
| Dosage | entry | `dosage`, expected alternate allele count |

Performance guidance:

- Request only the needed `entry_fields`.
- Use `variants=` to restrict import to selected variants when possible.
- Keep BGEN indexes stable and avoid rebuilding them inside every analysis run.

## Interval and BED Files

Use `hl.import_locus_intervals` for interval-list style files and `hl.import_bed` for UCSC BED. The reference genome must match the target MatrixTable's `locus` reference.

```python
intervals = hl.import_locus_intervals("targets.interval_list", reference_genome="GRCh38")
mt = mt.filter_rows(hl.is_defined(intervals[mt.locus]))
```

For string intervals in Python code:

```python
intervals = [hl.parse_locus_interval("1:100000-200000", reference_genome="GRCh37")]
mt = hl.filter_intervals(mt, intervals)
```

Use table joins for large interval sets; use collected Python lists only for small interval lists.

## Reference and Contig Conventions

Hail's `tlocus` includes a reference genome. Common mismatches:

| Data convention | Likely reference option | Typical fix |
| --- | --- | --- |
| `1`, `2`, ..., `X`, `Y`, `MT` | `GRCh37` | Import with `reference_genome="GRCh37"` |
| `chr1`, `chr2`, ..., `chrX`, `chrY`, `chrM` | `GRCh38` | Import with `reference_genome="GRCh38"` or recode labels deliberately |
| Numeric PLINK sex chromosomes `23`, `24`, `25`, `26` | human references | Rely on or customize PLINK `contig_recoding` |
| Same contig names but different assembly | ambiguous | Do not use `contig_recoding`; use liftover or correct source data |
| Non-model organism or custom contigs | custom `ReferenceGenome` or `reference_genome=None` | Define reference lengths or use untyped locus structs |

`contig_recoding` changes contig labels during import. It does not transform positions, validate allele strand, or lift between coordinate systems.

## Common MatrixTable Field Layouts

### VCF-Derived Dense Calls

```text
row key:    locus: locus<GRCh37/GRCh38>, alleles: array<str>
row:        rsid: str, qual: float64, filters: set<str>, info: struct{...}
col key:    s: str
entry:      GT: call, AD: array<int32>, DP: int32, GQ: int32, PL: array<int32>, ...
global:     optional metadata or CSQ header
```

### PLINK-Derived Calls

```text
row key:    locus, alleles
row:        rsid, cm_position
col key:    s
col:        fam_id, pat_id, mat_id, is_female, is_case or quant_pheno
entry:      GT: call
```

### BGEN-Derived Dosages

```text
row key:    locus, alleles
row:        varid and/or rsid when requested
col key:    s
entry:      dosage: float64 and/or GP: array<float64> and/or GT: call
```

### Post-QC Dense Calls

```text
row:        variant_qc: struct{AC, AF, AN, call_rate, p_value_hwe, ...}
col:        sample_qc: struct{call_rate, n_called, r_ti_tv, dp_stats, ...}
entry:      GT plus filtered/updated entry fields
```

### Split Multiallelic Calls

```text
row key:    locus, alleles  # now biallelic rows for split records
row:        a_index: int32, was_split: bool, info: struct{...}
entry:      downcoded GT and adjusted HTS fields
```

## MatrixTable vs Table Decision

Use a `MatrixTable` when the core data has sample-by-variant entries. Use a `Table` when records are one-dimensional:

- Variant annotation catalogs: `Table` keyed by `locus`, `alleles`.
- Sample phenotypes/covariates: `Table` keyed by `s`.
- Intervals: interval-keyed `Table`.
- Association results: row-keyed `Table`.
- Related pairs: pair-keyed `Table`.

Avoid `mt.entries()` unless a downstream operation genuinely needs one row per sample-variant pair. It expands data size by `n_rows * n_cols`.
