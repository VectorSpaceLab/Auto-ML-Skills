# Message Formats

Pydantic AI represents conversation state as provider-independent dataclasses in `pydantic_ai.messages`. Use this reference when persisting history, trimming messages, replaying a conversation, or translating between app/UI protocol messages and Pydantic AI messages.

## Message Model

`ModelMessage` is a discriminated union of:

- `ModelRequest`: application/user side messages sent to a model.
- `ModelResponse`: model side messages returned by a provider.

`ModelRequest` includes `parts`, optional `timestamp`, rendered `instructions`, `run_id`, `conversation_id`, and app-only `metadata`. `ModelResponse` includes response `parts`, usage, model/provider metadata, response IDs, finish reason, state, `run_id`, `conversation_id`, and app-only `metadata`.

Result objects expose history helpers:

- `result.all_messages()` returns prior and current messages.
- `result.new_messages()` returns only messages produced by the current run.
- `result.all_messages_json()` and `result.new_messages_json()` serialize directly to JSON bytes.

For streamed runs, result messages are complete only after output streaming finishes through `stream_output()`, `stream_text()`, `stream_response()`, or `get_output()`. If `.stream_text(delta=True)` is used, the final result content is not built as one message part.

## Request Parts and User Content

Common `ModelRequestPart` variants:

| Part | Use |
| --- | --- |
| `SystemPromptPart` | System/developer prompt content generated by the app. |
| `UserPromptPart` | User prompt content, including multimodal lists. |
| `ToolReturnPart` | Result for a user-defined function tool call. |
| `NativeToolReturnPart` | Result for a provider-native tool call. |
| `RetryPromptPart` | Retry instruction after validation/tool/output failure. |
| `InstructionPart` | Structured instruction content before rendering. |

`UserPromptPart.content` can be a string or a sequence of `UserContent` items. `UserContent` includes:

- `str` for simple text;
- `TextContent(content=..., metadata=...)` when app metadata should be preserved but not sent to the model;
- `ImageUrl`, `AudioUrl`, `VideoUrl`, `DocumentUrl` for URL-based files;
- `BinaryContent(data=..., media_type=...)` for local bytes;
- `UploadedFile(file_id=..., provider_name=...)` for provider-hosted files;
- `CachePoint(ttl='5m' | '1h')` for prompt-cache boundaries on supported providers.

For URL/file parts, route provider support and model choice to `../models-and-providers/`. This sub-skill covers the portable message objects and history boundaries.

## Response Parts

Common `ModelResponsePart` variants:

| Part | Use |
| --- | --- |
| `TextPart` | Plain text model response. |
| `ThinkingPart` | Provider thinking/reasoning content that may need provider-specific round-trip fields. |
| `CompactionPart` | Provider-produced summary/compaction content; round-trip to the same provider. |
| `FilePart` | Model-produced binary file, represented by `BinaryContent`/`BinaryImage`. |
| `ToolCallPart` | Call to a user-defined function or output tool. |
| `NativeToolCallPart` | Call to a provider-native tool. |

`ModelResponse` convenience properties such as `.text`, `.thinking`, `.files`, `.images`, `.tool_calls`, and `.native_tool_calls` help inspect response content without manual part filtering.

`ToolCallPart.args` may be a `dict`, JSON string, or `None`. Use `.args_as_dict()` or `.args_as_json_str()` when adapting messages to providers or diagnostics. Malformed JSON can be preserved as `{'INVALID_JSON': raw}` so retry flows can round-trip without crashing.

## Multimodal Content

Use these public content classes:

| Class | Typical input | Notes |
| --- | --- | --- |
| `ImageUrl`, `AudioUrl`, `VideoUrl`, `DocumentUrl` | Provider downloads or Pydantic AI downloads a URL depending on provider support and `force_download`. | Explicit `media_type` avoids inference failures on extensionless URLs. |
| `BinaryContent` | App reads bytes and passes a media type. | `from_path()` infers media type, defaulting unknown files to `application/octet-stream`; `from_data_uri()` requires base64 data URI syntax. |
| `BinaryImage` | Image bytes where image output/input type matters. | Requires `media_type` beginning with `image/`. |
| `UploadedFile` | Provider file ID or provider file URL. | Must carry a matching `provider_name`; uploaded file references are not portable across providers. |
| `TextContent` | Text with app metadata. | `metadata` is preserved in messages but not sent to the model. |

Media pitfalls:

- If a URL lacks an inferable extension, pass `media_type` explicitly.
- `BinaryContent.format` can raise `ValueError` for unsupported/unknown media types; use a standard media type when provider adapters need a format.
- `UploadedFile` histories should be replayed only with a compatible provider or rewritten/removed by history processing.
- If a tool returns multimodal content, `ToolReturnPart.files`, `.content_items(mode='str' | 'jsonable')`, and `.model_response_str()` help separate files from serializable data.
- If passing a file as user content and the model needs to refer to it later in tool arguments, include your own text label such as `"This is file <identifier>:"` before the file content; automatic identifiers are inserted for tool returns, not ordinary user prompts.

## Serializing and Loading History

Use `ModelMessagesTypeAdapter` for durable storage and cross-process handoff:

```python
from pydantic_ai import ModelMessagesTypeAdapter

json_bytes = ModelMessagesTypeAdapter.dump_json(messages)
restored = ModelMessagesTypeAdapter.validate_json(json_bytes)
```

For JSON-compatible Python objects:

```python
python_data = ModelMessagesTypeAdapter.dump_python(messages, mode='json')
restored = ModelMessagesTypeAdapter.validate_python(python_data)
```

The adapter preserves `run_id`, `conversation_id`, metadata, multimodal discriminators such as `kind: 'image-url'`, and binary data using base64-compatible serialization. It also supports backwards-compatible loading of older histories with missing newer fields.

When continuing a conversation, pass a sequence of `ModelMessage` to `message_history=`. Pydantic AI stamps fresh runs with a `run_id` and carries a `conversation_id` across runs that reuse history. Pass `conversation_id='new'` to fork a thread intentionally, or pass an app-owned conversation ID to align traces with your storage key.

## Trimming and Replaying History

When reducing history for cost or privacy, preserve the semantic adjacency needed by provider APIs:

- Keep every `ToolCallPart` with its matching `ToolReturnPart` unless both are intentionally removed.
- Do not leave a trailing `ModelResponse` containing unresolved tool calls unless the next run receives corresponding deferred results or the adapter deliberately drops them.
- Preserve provider-specific `ThinkingPart`, `CompactionPart`, `provider_name`, `provider_details`, and `UploadedFile` fields when replaying to the same provider; remove or rewrite them when changing providers if unsupported.
- Preserve the initial instructions/system prompt or use the `ReinjectSystemPrompt` capability when history came from a source that does not round-trip system prompts.
- When a router output function hands off to another agent, remove the final output-tool call from the source history before passing it to the next agent.

`ProcessHistory` can intercept and modify message history before each model request. It is useful for filtering, recent-message windows, and summarization. A context-aware processor can accept `RunContext` in addition to the message list. If a processor rebuilds the trailing request, preserve its parts/timestamp/instructions/metadata so `new_messages()` can still distinguish prior context from current-run messages.

## UI Adapter Boundaries

UI adapters such as AG-UI and Vercel AI translate frontend protocol messages into `Agent.run_stream_events()` inputs and translate Pydantic AI events back out. Keep detailed adapter implementation in `../mcp-and-integrations/` or `../cli-and-apps/`, but remember these message-boundary rules:

- Treat client-submitted history as untrusted unless it comes from server-side persistence.
- UI adapters strip client-submitted system prompts by default and replace them with the agent's configured prompt.
- Dangling client-submitted tool calls are dropped unless explicit deferred tool results resolve them.
- File URL schemes default to `http`/`https`; non-HTTP cloud schemes and unsafe `force_download` values are sanitized by default.
- Uploaded file references from clients are dropped by default unless the backend opts into preserving file data.

For stricter integrity, persist authoritative `ModelMessage` history server-side and pass that history into the adapter/run boundary rather than trusting a browser to resubmit assistant/tool state.
