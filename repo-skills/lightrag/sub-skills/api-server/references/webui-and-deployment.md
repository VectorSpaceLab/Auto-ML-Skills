# WebUI And Deployment

LightRAG ships a React 19 + TypeScript WebUI built with Vite and Bun. The installed API package can serve packaged WebUI assets from the API server at `/webui/`; source checkouts can rebuild those assets when developing or packaging.

## WebUI Commands

Run these from a source checkout that contains the `lightrag_webui` directory:

| Task | Command |
| --- | --- |
| Install frontend dependencies | `bun install --frozen-lockfile` |
| Start Vite dev server | `bun run dev` |
| Start Vite through Bun runtime | `bun run dev:bun` |
| Build production assets | `bun run build` |
| Build through Bun runtime | `bun run build:bun` |
| Preview built assets | `bun run preview` |
| Preview through Bun runtime | `bun run preview:bun` |
| Lint WebUI source | `bun run lint` |
| Run frontend unit tests | `bun test` or `bun run test` |
| Watch tests | `bun run test:watch` |
| Coverage | `bun run test:coverage` |

Bun is required for the project’s frontend tests. The build and lint scripts are ordinary npm scripts and can also run through Node package managers when Bun is unavailable:

```bash
npm install
npm run build
npm run lint
```

## Packaged WebUI Assets

Production builds write static assets under the API package’s WebUI asset directory, and the Python package includes those files as package data. The API server detects whether packaged assets exist and reports `webui_available` from `/health`.

Typical source-checkout build flow:

```bash
cd lightrag_webui
bun install --frozen-lockfile
bun run build
```

Then start the API server from the directory containing the desired `.env`:

```bash
lightrag-server
```

Open the WebUI at `http://localhost:9621/webui/` unless a reverse-proxy prefix changes the browser-visible URL.

## Runtime Path Prefixes

The WebUI does not need one build per deployment prefix. The FastAPI server injects runtime config into the served `index.html` containing:

- `apiPrefix`: browser-visible API prefix from `LIGHTRAG_API_PREFIX` or `--api-prefix`
- `webuiPrefix`: the composed WebUI mount prefix, usually `<apiPrefix>/webui/`

Frontend code reads those values at runtime and normalizes them before issuing API calls. In Vite dev mode, equivalent dev-time injection can be provided through `VITE_DEV_API_PREFIX` and `VITE_DEV_WEBUI_PREFIX` so path-prefix behavior matches production.

Reverse-proxy rule of thumb: the proxy may expose `/site01/...` to browsers while forwarding natural app paths to the backend. Do not bake `/site01` into REST route paths in client code; set `LIGHTRAG_API_PREFIX=/site01` and let FastAPI root-path plus WebUI runtime config handle browser URLs.

## WebUI Feature Map

The WebUI talks to the same route families documented in [api-routes](api-routes.md):

| WebUI area | Route families |
| --- | --- |
| Login/auth status | `/auth-status`, `/login` |
| Status cards and queue state | `/health`, `/documents/pipeline_status` |
| Document upload/management | `/documents/upload`, `/documents/paginated`, `/documents/track_status/{track_id}`, document delete/clear/reprocess routes |
| Retrieval view | `/query`, `/query/stream`, `/query/data` |
| Graph viewer/editor | `/graphs`, `/graph/label/*`, `/graph/entity/*`, `/graph/relation/*`, `/graph/entities/merge` |

Supported upload extensions in the WebUI include common text/source formats plus `pdf`, `docx`, `pptx`, and `xlsx`. Parser behavior and document-processing details belong to `../../document-pipeline/SKILL.md`.

## Docker Startup Paths

For a basic Docker Compose deployment with an already prepared `.env`:

```bash
docker compose up -d
```

For a stack generated by the interactive setup wizard:

```bash
docker compose -f docker-compose.final.yml up -d
```

Before exposing a stack beyond localhost, run:

```bash
make env-security-check
```

Deployment boundaries:

- The base `.env` remains host-usable.
- Wizard-generated `docker-compose.final.yml` contains service-only rewrites such as container hostnames and mounted certificate paths.
- Rerun `make env-base-rewrite` or `make env-storage-rewrite` to repair wizard-managed compose service blocks from bundled templates.
- If the generated stack includes services that require external environment values at compose startup, export or set those variables before `docker compose`.

## Offline And Layered Installation Notes

The API server itself requires the API dependency extra when installing from Python packaging:

```bash
pip install 'lightrag-hku[api]'
```

Layer optional extras only when the deployment needs them:

| Extra | Adds |
| --- | --- |
| `api` | FastAPI/Uvicorn/Gunicorn, auth, multipart upload, document processing dependencies, WebUI-serving support. |
| `offline-storage` | Storage backend clients such as Redis, PostgreSQL, Milvus, Qdrant, MongoDB, OpenSearch, Faiss, and Neo4j dependencies. |
| `offline-llm` | Additional LLM/embedding provider client libraries. |
| `offline` | API plus offline storage and offline LLM extras. |

Offline image/runtime caveats:

- High-compute document extraction and local model stacks may require packages or model weights that are not preinstalled in standard images.
- Do not assume `transformers`, `torch`, CUDA runtimes, Docling local execution, or model caches are present unless the deployment explicitly prepared them.
- Use hosted providers or separately managed local services when a minimal/offline server image lacks model/runtime dependencies.

## WebUI Testing Caveats

For source-level frontend work:

- Use Bun’s built-in test runner, not Vitest or Jest.
- Vite dev servers keep long-lived polling connections, so browser automation should prefer `domcontentloaded` plus a selector wait instead of `networkidle`.
- WebUI state uses browser storage; for UI-only debugging, seed settings in local storage rather than triggering live model calls.
- If the API is reverse-proxied with `LIGHTRAG_API_PREFIX`, test both `/webui/` in direct backend mode and the prefixed browser URL behind the proxy.
