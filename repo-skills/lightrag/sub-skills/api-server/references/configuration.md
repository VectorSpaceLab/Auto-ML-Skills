# API Server Configuration

LightRAG API configuration is environment-first. The server loads `.env` from the current startup directory, then lets already-exported OS environment variables take precedence. Command-line flags override the same logical settings for that process.

## Entry Points

| Command | Purpose | Notes |
| --- | --- | --- |
| `lightrag-server` | Start the FastAPI app with Uvicorn. | Single-process mode; if `--workers` is greater than 1, Uvicorn mode forces it back to one worker. |
| `lightrag-gunicorn --workers 4` | Start Gunicorn with Uvicorn workers. | Production-style multiprocess mode; not a Windows path. |
| `lightrag-hash-password` | Generate bcrypt password values for `AUTH_ACCOUNTS`. | Use `--username USER PASSWORD` to emit a ready `USER:{bcrypt}...` entry. |

Common startup flags:

```bash
lightrag-server --host 0.0.0.0 --port 9621 --working-dir ./rag_storage --input-dir ./inputs
lightrag-server --api-prefix /site01 --log-level INFO
lightrag-gunicorn --workers 4 --timeout 150
```

Use the bundled check script before operational work when you only need to confirm that the installed package exposes the expected API console scripts:

```bash
python scripts/check_api_entrypoints.py
```

## Core Server Settings

| Env var / flag | Meaning | Default shape |
| --- | --- | --- |
| `HOST` / `--host` | Bind host. | `0.0.0.0` |
| `PORT` / `--port` | Bind port. | `9621` |
| `WORKING_DIR` / `--working-dir` | Persistent RAG data directory. | `./rag_storage` |
| `INPUT_DIR` / `--input-dir` | Uploaded-file input directory. | `./inputs` |
| `TIMEOUT` / `--timeout` | Server/LLM request timeout used by server mode. | Project default, commonly `150` seconds |
| `WORKERS` / `--workers` | Gunicorn worker count. | `1` |
| `LOG_LEVEL` / `--log-level` | Logging level. | `INFO` |
| `VERBOSE` / `--verbose` | Extra debug output when logging is debug-level. | `false` |
| `LIGHTRAG_API_PREFIX` / `--api-prefix` | Reverse-proxy prefix visible to browsers. | Empty string |
| `CORS_ORIGINS` | Comma-separated CORS allowlist or `*`. | `*` |
| `WEBUI_TITLE`, `WEBUI_DESCRIPTION` | WebUI display text exposed by status/config. | Optional |
| `SSL`, `SSL_CERTFILE`, `SSL_KEYFILE` | Direct HTTPS mode. | Disabled |
| `MAX_UPLOAD_SIZE` | Upload size limit in bytes; `None` means unlimited. | `104857600` |

`LIGHTRAG_API_PREFIX` is passed to FastAPI as `root_path`. Routes remain mounted at natural paths inside the app (`/docs`, `/query`, `/documents`, `/webui`), while reverse-proxy clients can reach prefixed URLs such as `/site01/webui/` and OpenAPI advertises the prefixed server URL.

## Auth And Access Control

LightRAG combines three access patterns:

| Mechanism | Configure with | How clients send it | Typical use |
| --- | --- | --- | --- |
| JWT login | `AUTH_ACCOUNTS`, `TOKEN_SECRET`, `JWT_ALGORITHM`, token lifetime vars | `Authorization: Bearer <token>` after `POST /login` | WebUI and user login flows |
| API key | `LIGHTRAG_API_KEY` or `--key` | `X-API-Key: <key>` | Automation and scripts |
| Whitelist | `WHITELIST_PATHS` | No credential on matched paths | Health checks and selected compatibility endpoints |

Important auth rules:

- `AUTH_ACCOUNTS` is comma-separated `username:password` pairs. Passwords may be plaintext, but bcrypt values generated by `lightrag-hash-password` are safer.
- If `AUTH_ACCOUNTS` is configured, `TOKEN_SECRET` must be set and must not equal the built-in default guest-mode secret.
- `JWT_ALGORITHM` defaults to `HS256`; `none` is rejected.
- With no `AUTH_ACCOUNTS`, the server creates guest JWTs for WebUI use and logs a warning about the default guest-mode secret.
- `TOKEN_AUTO_RENEW=true` issues a refreshed JWT in the `X-New-Token` response header when a token is near expiry, except on high-frequency paths such as health and document status polling.
- `WHITELIST_PATHS` defaults to `/health,/api/*`; keep it narrow before exposing the service.

Generate bcrypt account values:

```bash
lightrag-hash-password --username admin 'replace-with-a-strong-password'
```

Use the emitted value in `.env`:

```bash
AUTH_ACCOUNTS='admin:{bcrypt}...'
TOKEN_SECRET='replace-with-a-long-random-secret'
JWT_ALGORITHM=HS256
```

## Query Defaults Exposed By The Server

These values affect REST query defaults and health/status reporting:

| Env var | Meaning |
| --- | --- |
| `TOP_K` | Default entity/relation retrieval count. |
| `CHUNK_TOP_K` | Default chunk retrieval count after reranking. |
| `MAX_ENTITY_TOKENS`, `MAX_RELATION_TOKENS`, `MAX_TOTAL_TOKENS` | Context budget controls. |
| `COSINE_THRESHOLD` | Similarity threshold for vector retrieval. |
| `RELATED_CHUNK_NUMBER` | Related chunks pulled around graph evidence. |

## Setup Wizard Outputs

Prefer the Make targets over invoking setup internals directly:

| Target | Use when |
| --- | --- |
| `make env-base` | First run: choose LLM, embedding, reranker, and optional local model services. |
| `make env-storage` | Add or change storage backends and database services after base setup. |
| `make env-server` | Configure port, auth, API key, SSL, and security-sensitive server settings. |
| `make env-validate` | Validate the current `.env`. |
| `make env-security-check` | Audit missing auth, weak JWT settings, unsafe whitelist paths, and placeholders. |
| `make env-backup` | Create a standalone `.env` backup. |
| `make env-base-rewrite`, `make env-storage-rewrite` | Force regeneration of wizard-managed service blocks. |

Wizard output rules:

- `.env` remains host-usable and is the active server configuration for the latest wizard run.
- Existing values are reused as defaults on reruns, and backups are made before the wizard overwrites `.env`.
- `docker-compose.final.yml` is created only when wizard-managed Docker services are selected or an existing generated stack must stay aligned.
- Container-only hostnames, service-internal URLs, and staged certificate paths belong in `docker-compose.final.yml` service overrides, not in host-facing `.env` values.
- Start a wizard-managed stack with `docker compose -f docker-compose.final.yml up -d`.

## `config.ini` Compatibility

`config.ini` is deprecated and kept only as a temporary compatibility example for backend connection sections such as `[neo4j]`, `[mongodb]`, `[redis]`, `[qdrant]`, `[postgres]`, `[memgraph]`, and `[milvus]`. Prefer `.env` or process environment variables for new API server work. If a task finds both styles, migrate operational guidance to `.env` and route backend-specific translation details to `../../storage-backends/SKILL.md`.
