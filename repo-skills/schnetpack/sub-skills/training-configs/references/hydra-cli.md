# SchNetPack Hydra CLI

SchNetPack exposes training and prediction through Hydra-backed console scripts:

- `spktrain` uses the `train` config and starts from defaults for `run`, `globals`, `trainer`, `callbacks`, `task`, `model`, `data`, `logger`, and `experiment`.
- `spkpredict` uses the `predict` config and requires a trained model directory plus prediction input/output paths.

Use CLI composition for common work; create project-local config files only when the command becomes too large or needs versioned reuse.

## Safe inspection commands

These commands should print help/resolved configuration without starting training or prediction:

```bash
spktrain --help
spktrain experiment=qm9_atomwise --help
spktrain experiment=md17 data.molecule=uracil --help
spkpredict --help
```

`spktrain experiment=... --help` is the safest way to confirm whether an override path exists before launching a run.

## Override syntax

Hydra uses two different override styles:

| Intent | Syntax | Example |
| --- | --- | --- |
| Set an existing scalar/list/dict field | `group.field=value` | `data.batch_size=64` |
| Set a nested object argument | `group.object.field=value` | `model.representation.n_interactions=5` |
| Replace an existing config group | `group/name=option` | `model/representation=schnet` |
| Add a missing key or nullable group | `+key=value` | `+matmul_precision=high` |
| Add or replace a callback/logger entry when it is absent | `+callbacks/name=option` or `callbacks/name=option` depending on current defaults | `callbacks/ema=ema` |

Rule of thumb: use `/` when selecting a YAML option from a config group, and use `.` when editing a field inside the already-composed config.

## Core config groups

`spktrain` starts from these groups:

| Group | Typical role | Useful options or fields |
| --- | --- | --- |
| `experiment` | Whole experiment template that overrides defaults | `qm9_atomwise`, `qm9_dipole`, `md17`, `rmd17`, `response` |
| `data` | Lightning data module or benchmark dataset | `custom`, `qm9`, `md17`, `rmd17`, plus dataset-specific fields |
| `model` | `NeuralNetworkPotential` template | `nnp` |
| `model/representation` | Representation subconfig | `painn`, `schnet`, `field_schnet`, `so3net` if available in the installed package |
| `task` | Lightning task, optimizer, scheduler, outputs | `default_task`, `task/optimizer=adam`, `task/optimizer=sgd`, `task/scheduler=reduce_on_plateau` |
| `trainer` | PyTorch Lightning trainer | `default_trainer`, `debug_trainer`, `ddp_trainer`, `ddp_debug` |
| `callbacks` | Lightning callbacks | `checkpoint`, `earlystopping`, `lrmonitor`, `ema` |
| `logger` | Logger dict passed to trainer | `tensorboard`, `csv`, `aim`, `wandb` |
| `run` | Working/data/run/checkpoint paths | `run.data_dir`, `run.path`, `run.id`, `run.ckpt_path` |
| `globals` | Shared interpolated variables | `globals.cutoff`, `globals.lr`, `globals.model_path`, property keys |

The `experiment` group is often the right starting point because the base train config has `model: null`, `data: custom`, and missing task outputs until an experiment or custom config fills them.

## Project-local configs

`spktrain` searches the current working directory and `./configs` in addition to the package configs. A project can define a custom experiment at either:

```text
./experiment/my_experiment.yaml
./configs/experiment/my_experiment.yaml
```

Then run:

```bash
spktrain experiment=my_experiment --help
spktrain experiment=my_experiment run.data_dir=/path/to/data run.path=/path/to/runs run.id=my-run
```

For an explicit config root:

```bash
spktrain --config-dir=/path/to/configs experiment=my_experiment --help
```

The config directory is the root containing group subdirectories like `experiment/`, `data/`, or `model/representation/`; do not point `--config-dir` at the `experiment/` subdirectory itself.

## Path side effects

Training enables `hydra.job.chdir=true`, and the Hydra run directory is `${run.path}/${run.id}`. SchNetPack writes a `config.yaml` into the run directory and stores checkpoints in `checkpoints/` relative to that run directory. Avoid surprises by setting all three path controls explicitly for reproducible commands:

```bash
spktrain experiment=qm9_atomwise \
  run.data_dir=/abs/or/project/data \
  run.path=/abs/or/project/runs \
  run.id=qm9-painn-cutoff5
```

The default `run.id` is generated by a UUID resolver, so repeated commands create different output directories unless `run.id` is set.

## Bounded smoke patterns

Do not use benchmark downloads or long training as a syntax check. If the user explicitly wants a tiny execution check on already prepared local data, bound it aggressively:

```bash
spktrain experiment=qm9_atomwise \
  run.data_dir=/path/to/existing/data \
  run.path=/tmp/spk-runs \
  run.id=syntax-fast-dev \
  trainer.fast_dev_run=true \
  data.num_workers=0 data.num_val_workers=0 data.num_test_workers=0
```

For custom data, also set tiny split counts, e.g. `data.num_train=8 data.num_val=2 data.batch_size=2`, but route schema/unit preparation to the data-pipelines sub-skill.
