# DiffDock UI Output Zip Format

## Archive Creation

The Gradio app runs inference in a temporary output directory and returns a downloadable zip named like:

```text
diffdock_output_YYYYMMDD_HHMMSS_<uuid>.zip
```

The archive is made from the full temporary inference output tree. In normal single-complex runs, DiffDock creates one complex-specific subdirectory inside that tree; the app also copies the input protein PDB into that subdirectory before zipping when exactly one output subdirectory exists.

Do not assume every archive has the same top-level directory name. Inspect by file extension and ranked SDF naming rather than hard-coding paths.

## Files The UI Looks For

The app extracts display data by scanning every non-directory zip member:

- Members ending in `.pdb` are read as protein text.
- Members ending in `.sdf` are read as ligand-pose text.
- SDF files are sorted by parsed rank when possible.

The UI uses the first PDB file found as the visualization protein. Every SDF with a parsed confidence value can become one dropdown visualization entry.

## Ranked SDF Naming

The app's filename parser expects SDF basenames shaped like:

```text
rank1.sdf
rank1_confidence-0.10.sdf
rank2_confidence-1.25.sdf
```

Parsed fields:

- `rank`: integer after the `rank` prefix.
- `confidence`: `None` when the basename has only `rankN`; otherwise the floating-point value after `confidence`.

Important behavior:

- `rank1.sdf` is recoverable as an SDF pose but does not produce a visualization dropdown label because its confidence is missing.
- `rank1_confidence-0.10.sdf` produces a label like `Rank 1. Confidence -0.10`.
- Nonconforming SDF names may still contain useful structure text, but the original app parser can fail or produce incomplete metadata.

## Visualization Behavior

The app builds visualization entries only for confidence-labelled SDF files. For each labelled SDF, it combines:

- The first discovered PDB file's content, if present.
- The SDF file content.
- A 3Dmol.js iframe generated by the app's molecular viewer helper.

If no PDB file is present, the app keeps the default visualization message instead of a protein-ligand 3D view. If SDF files exist but all lack confidence labels, the output archive can still be useful, but the ranked-sample dropdown can be empty.

## Inspecting Archives With The Bundled Helper

Use the bundled helper to summarize a UI result zip without importing Gradio, DiffDock, RDKit, or Torch:

```bash
python scripts/inspect_diffdock_output_zip.py diffdock_output.zip
```

The helper prints JSON with:

- Archive path and member count.
- PDB entries with path, size, SHA-256, and content presence.
- SDF entries with path, parsed rank, parsed confidence, parse status, size, SHA-256, and content presence.
- Visualization-oriented booleans, including whether any PDB exists and whether any confidence-labelled SDF exists.
- Warnings for missing PDB files, missing SDF files, malformed ranked names, and SDF files without confidence labels.

The helper intentionally does not render molecules or validate chemistry. It is for archive triage and handoff to downstream tools.

## Recovery Guidance

When a returned zip has no browser visualization:

1. Inspect the zip with `scripts/inspect_diffdock_output_zip.py`.
2. If PDB and SDF content are present, recover those files from the archive and view them in another molecular viewer.
3. If SDF files are present but confidence labels are missing, explain that the app dropdown depends on confidence-labelled filenames; the structures may still be recoverable.
4. If PDB is missing, explain that protein-ligand visualization cannot be reconstructed by the app viewer from the archive alone, but SDF pose files may still be useful.
5. If neither PDB nor SDF files are present, inspect `output.log` or other archive logs for inference failures and route dependency/runtime issues to the root troubleshooting reference.
