# dbt, MLflow, OpenLineage, and DQM Integrations

This reference covers Feast integrations that sit around the feature store core. For feature definitions generated by these tools, route object modeling details to `../../feature-definitions/SKILL.md`; for retrieval or materialization after integration setup, route to `../../retrieval-and-materialization/SKILL.md`.

## dbt Import

Feast includes a `feast dbt` CLI group for importing dbt models from compiled `manifest.json` artifacts. The integration is an alpha-style workflow: treat generated Python as reviewable source, not as an opaque runtime dependency.

Install and preflight:

```bash
pip install 'feast[dbt]'
python scripts/check_optional_extra.py --extras dbt
feast dbt --help
```

Workflow:

1. Tag dbt models intended for features, commonly with a `feast` tag.
2. Run `dbt compile` or `dbt run` in the dbt project so `target/manifest.json` exists.
3. Inspect import candidates.
4. Generate Feast definitions to a Python file.
5. Review generated entities, sources, feature views, types, TTLs, and tags.
6. Apply via normal Feast repo workflow.

Commands:

```bash
feast dbt list target/manifest.json --tag-filter feast
feast dbt import target/manifest.json \
  --entity-column user_id \
  --data-source-type snowflake \
  --tag-filter feast \
  --output features_from_dbt.py
```

Supported data source targets for generated source objects are `bigquery`, `snowflake`, and `file`. The mapper creates Feast objects such as `Entity`, `FeatureView`, and source classes including `BigQuerySource`, `SnowflakeSource`, or `FileSource`. dbt metadata is preserved in tags like `dbt.model`, `dbt.unique_id`, and `dbt.tag.<tag>`.

Useful options:

| Option | Purpose |
| --- | --- |
| `--entity-column` / `-e` | Entity column that must exist in imported models; can be repeated for multi-entity models. |
| `--data-source-type` / `-d` | Source family: `bigquery`, `snowflake`, or `file`. |
| `--tag-filter` / `-t` | Import only models with the chosen dbt tag. |
| `--output` / `-o` | Write generated Python definitions to a file. |
| `--dry-run` | Preview generated output without writing. |

Validation commands after generation:

```bash
python -m py_compile features_from_dbt.py
feast validate
feast plan
```

Troubleshooting signals:

- `dbt manifest not found`: run `dbt compile` and point to `target/manifest.json`.
- Missing `dbt_artifacts_parser`: install `feast[dbt]`.
- Models skipped: verify `--tag-filter` and that every imported model contains each `--entity-column`.
- Unsupported source type: use one of `bigquery`, `snowflake`, or `file`, or manually adapt generated definitions.

## MLflow Integration

Feast can log feature usage metadata to MLflow and expose Feast-enhanced MLflow operations through `store.mlflow`. It is disabled unless configured.

Install and preflight:

```bash
pip install 'feast[mlflow]'
python scripts/check_optional_extra.py --extras mlflow
```

`feature_store.yaml`:

```yaml
project: model_training
provider: local
registry: data/registry.db
offline_store:
  type: dask
online_store:
  type: sqlite
  path: data/online_store.db
mlflow:
  enabled: true
  tracking_uri: http://mlflow.example.internal:5000
```

Verified config facts:

- `mlflow.enabled` defaults to `false`.
- `mlflow.tracking_uri` can be omitted; Feast resolves the effective URI from MLflow/environment behavior.
- `FeatureStore(repo_path=...)` exposes `store.mlflow` when enabled; when omitted or disabled, guard with `if store.mlflow:`.

Typical usage:

```python
from feast import FeatureStore

store = FeatureStore(repo_path=".")
if not store.mlflow:
    raise RuntimeError("Enable mlflow.enabled in feature_store.yaml")

with store.mlflow.start_run(run_name="training"):
    training_df = store.get_historical_features(
        entity_df=entity_df,
        features=["driver_stats:conv_rate"],
    ).to_df()
    store.mlflow.log_training_dataset(training_df, dataset_name="driver_training")
    store.mlflow.log_model(model, "model")
```

Common Feast-enhanced API names include `store.mlflow.start_run`, `log_model`, `register_model`, `load_model`, `resolve_features`, `get_training_entity_df`, `log_training_dataset`, `active_run_id`, and `client`.

Failure signals:

- `ImportError` naming MLflow: install `feast[mlflow]`.
- `RuntimeError` from module-level `feast.mlflow`: bind a configured `FeatureStore` or set `mlflow.enabled: true`.
- No logged feature metadata: ensure retrieval happens inside an active MLflow run and through the configured `FeatureStore`.

## OpenLineage Integration

Feast can emit OpenLineage events for feature workflows. `RepoConfig` accepts an `openlineage` block and converts it to an OpenLineage config.

Install and preflight:

```bash
pip install 'feast[openlineage]'
python scripts/check_optional_extra.py --extras openlineage
```

HTTP transport example:

```yaml
openlineage:
  enabled: true
  transport_type: http
  transport_url: http://marquez.internal:5000
  transport_endpoint: api/v1/lineage
  namespace: feast-production
```

File transport example for local smoke tests:

```yaml
openlineage:
  enabled: true
  transport_type: file
  namespace: feast-local
  log_file_path: openlineage_events.json
```

Config facts:

- `enabled` defaults to `true` in the OpenLineage config object when a block is present.
- `transport_type` may be `http`, `console`, `file`, `kafka`, or omitted to use OpenLineage SDK defaults.
- HTTP requires `transport_url`; default `transport_endpoint` is `api/v1/lineage`.
- Environment variables include `FEAST_OPENLINEAGE_ENABLED`, `FEAST_OPENLINEAGE_TRANSPORT_TYPE`, `FEAST_OPENLINEAGE_URL`, and `FEAST_OPENLINEAGE_NAMESPACE`.

Validation steps:

```bash
feast validate
python - <<'PY'
from feast import FeatureStore
store = FeatureStore(repo_path='.')
print(store.config.openlineage)
PY
```

Failure signals:

- Missing `openlineage.client`: install `feast[openlineage]`.
- HTTP config error: set `transport_url` when `transport_type: http`.
- No events: confirm integration is enabled, the client can reach the endpoint, and the operation emits lineage.

## Data Quality Monitoring and Great Expectations

Feast DQM integrates validation concepts with Feast feature workflows. The Great Expectations profiler path requires the `ge` extra.

Install and preflight:

```bash
pip install 'feast[ge]'
python scripts/check_optional_extra.py --extras ge
```

`RepoConfig` accepts a `data_quality_monitoring` block. Feature views and stream/batch feature views also expose `enable_validation` in their constructors. Current verified signatures include:

```python
FeatureView(..., enable_validation=False)
BatchFeatureView(..., enable_validation=False)
StreamFeatureView(..., enable_validation=False)
```

Great Expectations profiler shape:

```python
import pandas as pd
from great_expectations.core import ExpectationSuite
from feast.dqm.profilers.ge_profiler import GEProfiler


def build_suite(dataset: pd.DataFrame) -> ExpectationSuite:
    suite = ExpectationSuite(expectation_suite_name="driver_features")
    # Add expectations with Great Expectations APIs.
    return suite

profiler = GEProfiler(user_defined_profiler=build_suite)
```

Use DQM for validation/reporting design. If the user asks how to retrieve validated rows or materialize features, route to `../../retrieval-and-materialization/SKILL.md`.

Failure signals:

- `ModuleNotFoundError: great_expectations`: install `feast[ge]`.
- Validation not running: confirm `enable_validation=True` on the relevant feature view and that a validation reference/profile is applied in the normal Feast workflow.
- Schema mismatch: compare source columns to declared `Field` schema and entity join keys.

## Integration Review Checklist

Before handing off an integration plan:

- State the minimum extras, not the broad dev/CI extra set.
- Include the exact YAML block and where it belongs in `feature_store.yaml`.
- Include a no-service preflight command.
- Identify credential/service prerequisites separately from Python import prerequisites.
- Route generated feature definition review to `../../feature-definitions/SKILL.md` and execution workflows to the appropriate sibling skill.
