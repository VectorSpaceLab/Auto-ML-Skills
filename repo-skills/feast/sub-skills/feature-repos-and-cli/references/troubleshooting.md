# Feature Repository and CLI Troubleshooting

## Install and Import Failures

Signals:

- `feast: command not found`
- `ModuleNotFoundError: No module named 'feast'`
- Optional backend errors for stores such as Redis, Snowflake, BigQuery, Spark, Milvus, or SQL registry.

Checks:

```bash
which feast
feast version
python -c "import feast; print(getattr(feast, '__version__', 'unknown'))"
python scripts/feature_repo_doctor.py feature_repo --require-feast
```

Fixes:

- Install Feast in the same Python environment that runs the CLI and notebooks.
- Install backend extras only for the store/provider in `feature_store.yaml`; do not install every optional dependency by default.
- If the CLI and Python import disagree, inspect `which feast`, `python -m pip show feast`, and shell environment activation.
- Optional store dependency selection belongs in `../../integrations-and-extensibility/SKILL.md`.

## Config Discovery Errors

Signal:

```text
Can't find feature repo configuration file at ... Make sure you're running feast from an initialized feast repository.
```

Likely causes:

- Running `feast` from the project root instead of the nested `feature_repo` generated by `feast init`.
- Supplying `--chdir` to the parent directory rather than the directory containing `feature_store.yaml`.
- Supplying a bad `--feature-store-yaml` path.
- Moving or renaming the config file without updating automation.

Fix:

```bash
find . -name feature_store.yaml -maxdepth 4
feast -c /correct/path/to/feature_repo configuration
python scripts/feature_repo_doctor.py /correct/path/to/feature_repo
```

## YAML and Config Validation

Signals:

- YAML parser errors such as `mapping values are not allowed here`.
- Pydantic validation errors from `RepoConfig`.
- Invalid provider, registry, online store, offline store, auth, or feature server type errors.

Fixes:

- Run `python scripts/feature_repo_doctor.py feature_repo` for static YAML shape and path hints.
- Keep `project`, `provider`, and `registry` at the top level.
- Use explicit `online_store.type` for non-default stores.
- For local workflows, start from a minimal known-good config and add store blocks one at a time.

Known-good local base:

```yaml
project: my_project
registry: data/registry.db
provider: local
online_store:
  type: sqlite
  path: data/online_store.db
entity_key_serialization_version: 3
```

## Project Mismatch

Signals:

```text
Project object name should match with the project name defined in feature_store.yaml
Multiple projects found in the repository. Currently no support for multiple projects
No project found in the repository. Using project name ... defined in feature_store.yaml
```

Fixes:

- If the repo defines a `Project(name=...)`, make it exactly match `project` in `feature_store.yaml`.
- If there are multiple project objects, split repos or remove extra project definitions.
- If no project object exists, Feast can use the YAML `project` value; this is acceptable for simple repos.

## Repo Parsing and Python Import Side Effects

Signals:

- `feast plan` or `feast apply` fails while importing an unrelated Python script.
- A helper script runs database, network, file deletion, or training code during `apply`.
- Errors mention module-level variables from files not intended as Feast definitions.

Cause:

Feast recursively imports Python files under the feature repo, excluding files matched by `.feastignore` and built-in ignored paths.

Fix:

```text
# .feastignore
scripts/*.py
notebooks/**/*.py
training_jobs
```

Then rerun:

```bash
feast -c feature_repo plan --skip-source-validation
```

## Source Validation Failures

Signals:

- Local file source path is missing.
- Remote table, bucket, topic, or stream is unavailable.
- Backend credentials fail during `plan` or `apply`.

Fixes:

- For local development, create or correct local data paths before `apply`.
- Use `--skip-source-validation` only for a reviewed dry plan when source availability is intentionally deferred.
- For production apply, remove the skip flag so Feast catches missing sources.
- For feature schema and data source object modeling, use `../../feature-definitions/SKILL.md`.

## Registry and Online Store Path Problems

Signals:

- Listing commands return empty objects after an apparently successful apply.
- A new `data/registry.db` appears in an unexpected directory.
- SQLite online store path fails because the parent directory does not exist.

Fixes:

```bash
feast -c feature_repo configuration
python scripts/feature_repo_doctor.py feature_repo
mkdir -p feature_repo/data
feast -c feature_repo registry-dump
```

Confirm that automation consistently uses the same `--chdir`, `--feature-store-yaml`, `project`, and registry path.

## CLI/API Misuse

Common mistakes:

- Passing both `config` and `fs_yaml_file` to `FeatureStore`; use only one.
- Using `FeatureStore(repo_path=...)` with a path that does not contain the selected YAML file.
- Expecting `FeatureStore.apply([...], partial=True)` to reconcile deletions for the whole repo. Use CLI `feast apply` for desired-state reconciliation.
- Using `plan` as a no-side-effect static command. It imports repo Python files.

Verified constructor:

```python
FeatureStore(repo_path=None, config=None, fs_yaml_file=None)
```

## Backend Credentials and Services

Signals:

- Provider login errors printed by `plan` or `apply`.
- SQL registry connection failures.
- Remote offline/online store import or credential errors.

Fixes:

- Confirm the selected provider and store blocks with `feast configuration`.
- Check environment variables, local auth files, network access, and optional Python extras for the selected backend.
- For server, remote registry, TLS, RBAC, and feature server auth problems, route to `../../servers-and-remote/SKILL.md`.

## When to Escalate to Other Sub-skills

- Feature definition constructor errors, duplicate feature view names, bad `Field` dtype, TTL, vector fields, or data source definitions: `../../feature-definitions/SKILL.md`.
- Historical retrieval, online retrieval, materialization windows, null online values, push ingestion, or saved datasets: `../../retrieval-and-materialization/SKILL.md`.
- Feature server, offline server, registry server, transformation server, MCP, TLS, auth, and remote topology: `../../servers-and-remote/SKILL.md`.
- Feast source tests, linting, docs, protos, Go, Java, or contributor workflows: `../../repo-development/SKILL.md`.
