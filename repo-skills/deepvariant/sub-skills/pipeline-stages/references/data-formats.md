# Data Formats and Path Semantics

This reference describes portable data contracts for reasoning about DeepVariant stage intermediates.

## Sharded File Specs

DeepVariant uses Nucleus sharded path syntax:

```text
logical spec:  prefix@N.suffix
shard files:   prefix-00000-of-000NN.suffix ... prefix-000NN-of-000NN.suffix
glob pattern:  prefix-?????-of-000NN.suffix
```

Examples:

| Spec | First shard | Last shard | Glob-style read pattern |
| --- | --- | --- | --- |
| `examples.tfrecord@10.gz` | `examples.tfrecord-00000-of-00010.gz` | `examples.tfrecord-00009-of-00010.gz` | `examples.tfrecord-?????-of-00010.gz` |
| `gvcf.tfrecord@32.gz` | `gvcf.tfrecord-00000-of-00032.gz` | `gvcf.tfrecord-00031-of-00032.gz` | `gvcf.tfrecord-?????-of-00032.gz` |
| `calls@100.tfrecord.gz` | `calls-00000-of-00100.tfrecord.gz` | `calls-00099-of-00100.tfrecord.gz` | `calls-?????-of-00100.tfrecord.gz` |

Rules:

- `@N` requires `N` to be a positive integer; `@0` and `@foo` are invalid sharded specs. Nucleus can parse suffixless specs such as `examples.gz@32`, but DeepVariant TFRecord writers usually expect the TFRecord-like extension to remain visible, as in `examples.tfrecord@32.gz`.
- The shard number width is at least five digits and grows for very large shard counts.
- `filename@N.gz` is not a file on disk; it is a logical collection used by DeepVariant readers and writers.
- Readers can often accept `@N`, comma-separated patterns, or glob-like forms; writers need either an unsharded output or a valid `@N` spec plus the correct per-task execution.
- Sharding consistency is resolved from the first filespec: if `--examples` is `@32`, paired `--gvcf` and side outputs must also be `@32`; if `--examples` is unsharded, paired outputs must be unsharded.

Use `scripts/sharded_path_helper.py` to expand and validate specs without reading data.

## TFRecord Payloads

DeepVariant stage intermediates are TFRecord containers, usually gzip-compressed when the path ends in `.gz`.

| Path role | Writer | Reader | Proto payload | Typical extension |
| --- | --- | --- | --- | --- |
| `--examples` | `make_examples` | `call_variants`, visualization utilities, training utilities | TensorFlow `tf.Example` containing encoded pileup image, variant/alt-allele metadata, and optional labels/debug fields. | `.tfrecord.gz` or `.tfrecords.gz` |
| `--gvcf` / `--nonvariant_site_tfrecord_path` | `make_examples` | `postprocess_variants` | Nucleus `Variant` protos for non-variant reference-confidence records. | `.tfrecord.gz` |
| `--outfile` from `call_variants` / `--infile` for `postprocess_variants` | `call_variants` | `postprocess_variants` | DeepVariant `CallVariantsOutput` protos with genotype likelihoods and candidate variant data. | `.tfrecord.gz` |
| candidate-position files | `make_examples --mode candidate_sweep` | later `make_examples` partitioning | Binary candidate-position arrays used for balanced partitioning. | implementation-specific binary path |

Do not treat these TFRecords as plain text. Inspecting records requires TensorFlow, Nucleus, DeepVariant protos, and compatible compiled dependencies; it is not a safe lightweight check.

## `.example_info.json`

`make_examples` writes example metadata using the examples prefix. For sharded `@N` specs, the metadata is written for the first shard path only.

The JSON records:

- DeepVariant version.
- Tensor shape of the encoded example image.
- Channel enum sequence or channel list.

`call_variants` and calling-mode `make_examples` use model metadata such as `model.example_info.json` or older `example_info.json` to validate input shape, channels, channel ablation, and model-specific flags. Mismatches usually mean the wrong model type, wrong custom model metadata, wrong `--channel_list`, or examples generated by a different DeepVariant/model configuration.

## Channel Lists

Default base channels are:

```text
read_base,base_quality,mapping_quality,strand,read_supports_variant,base_differs_from_ref
```

Additional channel names recognized in r1.10 include `haplotype`, `allele_frequency`, `insert_size`, `base_methylation`, `base_6ma`, `read_mapping_percent`, `avg_base_quality`, `identity`, `gap_compressed_identity`, `gc_content`, `is_homopolymer`, homopolymer quality channels, `mean_coverage`, `supplementary_alignment`, and `allele_sample_probability`. The symbolic `BASE_CHANNELS` expands to the six default channels in `--channel_list` contexts.

Rules:

- In calling mode with a checkpoint, r1.10 can derive channel configuration and selected calling flags from model metadata.
- If no model metadata supplies channels and the run is not `candidate_sweep`, `--channel_list` may be required for custom low-level commands.
- `--channels`, `--add_hp_channel`, and `--use_allele_frequency` are obsolete; use `--channel_list` entries such as `haplotype` or `allele_frequency` instead.
- Alt-aligned pileup channels are controlled by `--alt_aligned_pileup` behavior and are not ordinary user-set channel-list entries.
- Training or custom model workflows must preserve exact channel/shape compatibility between generated examples and checkpoints; route broader training decisions to `../training-custom-models/SKILL.md`.

## gVCF Dataflow

A final gVCF is not produced by `make_examples` directly. The data contract is:

1. `make_examples --mode calling --gvcf nonvariant.tfrecord@N.gz` emits non-variant `Variant` protos.
2. `call_variants` emits variant predictions as CallVariantsOutput records.
3. `postprocess_variants --nonvariant_site_tfrecord_path nonvariant.tfrecord@N.gz --gvcf_outfile sample.g.vcf.gz` merges variant and non-variant records.

DeepVariant gVCF non-variant records use `<*>` as the unspecified alternate allele and merge adjacent reference-confidence records with compatible genotype quality. `--gvcf_gq_binsize` adjusts quality binning. `--include_med_dp=true` adds MED_DP to gVCF output when supported.

Important constraints:

- gVCF generation is valid only from `make_examples --mode calling`.
- `--gvcf` and `--examples` should have matching shard counts.
- `postprocess_variants` requires both the non-variant TFRecord input and final gVCF output path.
- gVCF postprocessing can be slower and more memory-intensive than VCF-only postprocessing, especially for low-depth data with many small non-variant blocks.

## Candidate Positions

`candidate_sweep` mode is an advanced partitioning pre-step:

- It discovers candidate positions and writes them to `--candidate_positions` files.
- It is not a replacement for normal calling; it must be followed by `make_examples --mode calling` using the candidate positions.
- It is incompatible with `--proposed_variants`.
- In non-sweep modes, a provided candidate-position file affects partitioning rather than directly becoming VCF calls.

Candidate positions help balance intervals by candidate count. They are not user-editable VCFs and should not be confused with `--proposed_variants` for a VCF candidate importer.

## Input File Assumptions

`make_examples` relies on Nucleus/htslib-style genomics IO assumptions:

- FASTA passed to `--ref` must have `.fai`; uncompressed or bgzip-compressed references are supported by official runtime builds.
- BAM/CRAM passed to `--reads` must exist, be aligned to a compatible reference, be coordinate-sorted, and have an index such as `.bai` or `.crai`.
- CRAM normally uses `--ref` as its reference source in modern DeepVariant; container-visible local POSIX paths matter for CRAM reference access.
- `--regions`, `--exclude_regions`, BED, VCF, truth/confident files, and proposed variants must use contig names compatible with `--ref`.
- Training truth VCFs should be bgzipped and tabix-indexed; confident regions should be BED-like and reference-compatible.
- DeepVariant processes shared contigs between reference and reads; missing contigs can silently reduce the callable region if not reviewed.

## Final VCF and gVCF Outputs

`postprocess_variants` writes sorted VCF records using the FASTA contig order. For gzipped VCF outputs, indexing normally occurs after writing. If any reference contig is too long for standard tabix limits, CSI indexing may be selected.

Sample name precedence for final VCF/gVCF is:

1. CallVariantsOutput record sample name.
2. Non-variant site TFRecord sample name when gVCF input is present.
3. `--sample_name` fallback.
4. DeepVariant default sample name.

If users see an unexpected sample name, check BAM read groups, wrapper `--sample_name`, and whether empty examples/CVO records forced a fallback path.
