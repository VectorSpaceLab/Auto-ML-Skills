# Heatmap Workflows

CLAM attention heatmaps are generated by `create_heatmaps.py` from a YAML config. The workflow is intentionally heavier than evaluation: it opens WSIs, segments tissue, scans patches or ROI regions, extracts encoder features on the fly, applies a trained CLAM checkpoint, saves attention scores, renders heatmaps, and optionally exports high-attention or low-attention patches.

## Standard Heatmap Run

1. Prepare slides readable by OpenSlide and a process-list CSV with at least `slide_id`.
2. Put the YAML config under `heatmaps/configs/`.
3. Put the process-list CSV under `heatmaps/process_lists/` when `data_arguments.process_list` is a filename.
4. Confirm `model_arguments.ckpt_path`, `model_arguments.model_type`, `model_arguments.model_size`, `model_arguments.drop_out`, `model_arguments.embed_dim`, and `exp_arguments.n_classes` match the training checkpoint.
5. Confirm `encoder_arguments.model_name` matches the feature family used for training.
6. Run the static validator before launching WSI inference.
7. Launch heatmap generation and answer the printed confirmation prompt.

Example command:

```bash
CUDA_VISIBLE_DEVICES=0 python create_heatmaps.py --config_file config_template.yaml
```

The script argument is named `--config_file`. It defaults to `heatmap_config_template.yaml`, but the README examples may refer to `--config`; prefer the verified help/script name for the current checkout.

## Non-Interactive Prompt Handling

Before opening slides or loading checkpoints, `create_heatmaps.py` prints the resolved config and asks:

```text
Continue? Y/N
```

Use this prompt as a final safety check for output directories, checkpoint path, encoder, ROI, and overlap. For scripted runs, pipe an affirmative response only after validation:

```bash
printf 'Y\n' | CUDA_VISIBLE_DEVICES=0 python create_heatmaps.py --config_file cohort_heatmaps.yaml
```

If the script appears to hang immediately after printing config values, it is usually waiting for this prompt.

## Raw Outputs

Raw assets are saved under:

```text
<raw_save_dir>/<save_exp_code>/<label-or-group>/<slide_id>/
```

Typical raw files include:

- `<slide_id>_mask.pkl`: serialized segmentation mask used by the WSI object.
- `<slide_id>_mask.jpg`: segmentation visualization.
- `<slide_id>.h5`: patch-level features and coordinates computed on the first pass.
- `<slide_id>.pt`: tensor of patch features loaded for slide-level inference.
- `<slide_id>_blockmap.h5`: HDF5 with `attention_scores` and `coords` for the non-overlapping block map.
- `<slide_id>_blockmap.png`: rendered block map image.
- `<slide_id>_<overlap>_roi_<true-or-false>.h5`: overlapping or ROI attention scores and coordinates when `calc_heatmap` is true.
- `<raw_save_dir>/<save_exp_code>/config.yaml`: copy of the resolved config.

The script also writes a process summary CSV under `heatmaps/results/`, named from the process-list filename or `save_exp_code`. It includes bag size and `Pred_<class-index>` / `p_<class-index>` columns.

## Production Outputs

Rendered deliverables are saved under:

```text
<production_save_dir>/<save_exp_code>/<label-or-group>/
```

Typical production files include:

- Final heatmap image named with overlap, ROI flag, blur flag, reference-score flag, blank-canvas flag, alpha, visualization level, binarization flag, threshold, and extension.
- Optional original-slide image when `heatmap_arguments.save_orig: true`.
- Optional sampled patch PNGs under `<production_save_dir>/<save_exp_code>/sampled_patches/...`.

Raw outputs are for debugging, reproducibility, and downstream inspection. Production outputs are the rendered files normally shared with users.

## Custom Cohort With ROI Boxes

For a custom cohort where each slide needs a region of interest:

1. Set `heatmap_arguments.use_roi: true`.
2. Add process-list columns `x1`, `x2`, `y1`, and `y2` for every row with `process == 1`.
3. Keep `slide_id` values consistent with `data_arguments.slide_ext`; the script appends the extension if it is missing.
4. If slides live in multiple roots, set `data_arguments.data_dir` to a mapping and add the `data_arguments.data_dir_key` column to the process list.
5. From this sub-skill directory, validate before runtime:

```bash
python scripts/validate_heatmap_config.py /path/to/heatmaps/configs/cohort_heatmaps.yaml --process-list-root /path/to/heatmaps/process_lists
```

Then run from the CLAM working tree or an equivalent checkout containing `create_heatmaps.py`:

```bash
printf 'Y\n' | CUDA_VISIBLE_DEVICES=0 python create_heatmaps.py --config_file cohort_heatmaps.yaml
```

Expected outputs include the coarse whole-slide block map, an ROI-specific attention HDF5 named with `roi_True`, and final rendered production heatmaps named with `roi_1`.

## Top-K and Range Patch Sampling

Use `sample_arguments.samples` to export example patches by attention score:

- `mode: topk`: highest-attention patches.
- `mode: reverse_topk`: lowest-attention patches.
- `mode: range_sample`: random patches within a percentile score window.

Each sample item needs `sample: true`, a `name`, `k`, `mode`, and usually `seed`. For `range_sample`, include `score_start` and `score_end` as percentile fractions, such as `0.45` and `0.55` for mid-attention patches. The script converts scores to percentiles before sampling.

Sampling runs after attention scores and coordinates have been loaded from the block map, and exported patches are read directly from the WSI at `patching_arguments.patch_level` and `patching_arguments.patch_size`.

## CONCH or UNI Heatmaps

When the checkpoint was trained using CONCH or UNI features, heatmap inference must use the same encoder family:

- For CONCH v1, set `encoder_arguments.model_name: conch_v1` and `model_arguments.embed_dim: 512`, install CONCH, and set `CONCH_CKPT_PATH` in the runtime shell.
- For UNI v1, set `encoder_arguments.model_name: uni_v1` and `model_arguments.embed_dim: 1024`, and set `UNI_CKPT_PATH` in the runtime shell.
- For ResNet50-truncated, set `encoder_arguments.model_name: resnet50_trunc` and `model_arguments.embed_dim: 1024`.

A checkpoint trained on one encoder family should not be paired with another during heatmap inference, even if the model type and class count are correct.

## Relationship to Other CLAM Stages

- Use WSI preprocessing first when the task is to tune tissue segmentation, generate coordinate files, or inspect patch masks before heatmap-specific inference.
- Use feature extraction guidance when selecting ResNet50, UNI, or CONCH and understanding patch image size or encoder checkpoint requirements.
- Use training/evaluation guidance when choosing `model_type`, `model_size`, `drop_out`, `n_classes`, and checkpoint paths from trained experiments.
