# Model Extension Troubleshooting

Use this reference when a model, tokenizer, processor, pipeline, or test change behaves inconsistently after edits.

## `make fix-repo` Reverted My Changes

Likely causes:

- edited a `# Copied from ...` block directly
- edited generated output from a `modular_<model>.py` source
- auto mappings, docs TOCs, or docstrings were regenerated by repository utilities

Fix:

1. Identify whether the changed lines live in a copied block or generated file.
2. Edit the original copied-from source or `modular_<model>.py` instead.
3. Run `make fix-repo` again.
4. Inspect the diff for expected generated updates.

If the copy must intentionally diverge, remove or adjust the copied-from marker and explain the divergence.

## Modular Conversion Overwrites Modeling Files

Symptom: edits in `modeling_<model>.py` disappear after conversion or `make fix-repo`.

Fix:

- make durable changes in `src/transformers/models/<model>/modular_<model>.py`
- regenerate with `python utils/modular_model_converter.py <model>`
- compare with `python utils/check_modular_conversion.py --files src/transformers/models/<model>/modular_<model>.py`

Inspect generated modeling files only to debug converter output.

## Missing Auto Mappings

Symptoms:

- `AutoConfig.from_pretrained(...)` cannot resolve the config
- `AutoModelFor...` loads the wrong class or raises an unrecognized configuration error
- `AutoTokenizer`, `AutoProcessor`, or `AutoImageProcessor` fails despite the config existing
- auto tests fail with missing or duplicate mapping entries

Fix:

1. Confirm the config class has a unique `model_type`.
2. Confirm the serialized config uses the same `model_type`.
3. Check `CONFIG_MAPPING_NAMES` and relevant modeling/tokenizer/processor mappings.
4. Run:

```bash
python utils/check_auto.py
python utils/check_auto.py --fix_and_overwrite
make fix-repo
```

5. Run the matching tests in `tests/models/auto/`.

## Wrong `model_type`

Symptoms:

- local config saves as one key but auto mappings use another
- `AutoConfig` returns a parent or unrelated config
- tokenizer or processor dispatch follows another model family

Fix:

- set `model_type` once on the config class and keep it stable
- ensure it is unique across Transformers
- ensure auto mappings use the same key
- ensure test fixtures and tiny configs serialize the same key
- avoid changing `model_type` after public checkpoints exist unless maintainers approve a compatibility plan

## Tokenizer Registration Gaps

Symptoms:

- fast tokenizer loads when the slow tokenizer was expected, or the reverse
- `use_fast=False` fails
- save/load round trip loses tokenizer class metadata
- `AutoTokenizer.register` raises because an existing mapping is being overwritten

Fix:

- update slow and fast tokenizer mappings together when both classes exist
- test `AutoTokenizer.from_pretrained(..., use_fast=True)` and `use_fast=False`
- ensure tokenizer config files include expected class metadata
- do not register over a built-in tokenizer class unless the change is intentional and approved

## Processor or Image/Video Processor Gaps

Symptoms:

- `AutoProcessor` does not return the multimodal processor
- image or video processor loads but the pipeline receives the wrong argument names
- processor save/load loses nested tokenizer or image processor settings

Fix:

- update processor, image processor, video processor, or feature extractor mappings according to modality
- validate `processor.save_pretrained` and `AutoProcessor.from_pretrained` round trips
- check that the model forward signature matches processor output keys
- verify pipeline preprocessing uses the same keys as model tests

## Pipeline Registration Fails

Symptoms:

- `pipeline("new-task")` says the task is unknown
- task is known but rejects the model class
- custom pipeline parameters do not reach `preprocess`, `_forward`, or `postprocess`

Fix:

- register with `PIPELINE_REGISTRY.register_pipeline`
- include the correct `pipeline_class`, backend model class, default model metadata, and task type
- implement `_sanitize_parameters` to return `(preprocess_kwargs, forward_kwargs, postprocess_kwargs)`
- test with the public `pipeline(...)` function and a tiny model where possible

## Failing Slow Tests

Symptoms:

- fast tests pass but integration tests fail with real checkpoints
- outputs differ only on specific GPUs
- downloads or large fixtures fail in PR-style environments

Fix:

- mark real-checkpoint tests with `@slow`
- run them explicitly with `RUN_SLOW=1 pytest ...`
- use `cleanup` in setup/teardown for memory-sensitive tests
- use hardware-aware expectations for numerical tolerances when needed
- keep tiny CPU tests independent of network and large checkpoints

## Optional Dependency ImportError

Symptoms:

- importing `transformers` works
- importing a model class or running a model test raises an optional dependency `ImportError`
- lightweight inspection can read configs and mappings but not instantiate PyTorch classes

Fix:

- install the required backend when model execution is needed
- guard imports with availability helpers such as `is_torch_available()`
- decorate tests with `require_torch` or other `require_*` decorators
- do not import backend classes at module import time unless guarded

## Contribution Policy Block

Symptoms:

- missing issue approval
- issue belongs to another contributor
- another PR already implements the same model or mapping
- change is an isolated typo or busywork

Fix:

- stop coding toward PR-ready output
- report the missing coordination evidence or duplicate PR
- ask the human to obtain maintainer approval or choose a non-overlapping scope

## Debugging Order for New Ports

When a port produces wrong outputs, compare original and Transformers implementations in this order:

1. tokenization or preprocessing output
2. input IDs, masks, pixel values, audio features, or video tensors
3. embeddings
4. first layer input
5. first layer output
6. later layer outputs
7. final logits or generated tokens
8. save/load and conversion script behavior

For similar decoder models, first check rotary embeddings, attention masks, key/value cache shapes, norm placement, activation functions, tensor parallel plans, and generation config defaults.
