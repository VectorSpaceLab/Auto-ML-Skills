# Hard Negatives and Rerank Inputs

Hard negatives are non-relevant documents retrieved near the top of a first-stage ranking. Tevatron uses them either as embedded `negative_passages` in legacy train JSONL or as compact `negative_document_ids` that point into a separate corpus.

## Hard-Negative Requirements

For retriever training, the effective negative count per query should match the training group size:

- `train_group_size` defaults to `8`.
- One passage is positive, so a fully populated record usually needs at least `7` usable negatives per query.
- Positives, duplicates, missing corpus IDs, and answer-containing exact-match candidates reduce the usable count after mining.
- If fewer negatives are present, Tevatron samples with replacement. This avoids immediate failure but weakens contrastive diversity.
- Hard-negative rankings should be grouped by query and ordered from strongest to weakest candidate before sampling or conversion.

## Supported Training Shapes

### Embedded Legacy Shape

```json
{"query_id":"q1","query":"question","positive_passages":[{"docid":"d1","text":"positive"}],"negative_passages":[{"docid":"d2","text":"hard negative"}]}
```

Use this when each train row contains all passage text needed for training.

### Corpus-ID Shape

```json
{"query_id":"q1","query_text":"question","positive_document_ids":["d1"],"negative_document_ids":["d2","d3"]}
```

Use this when train rows should stay compact and a separate corpus JSONL provides `docid`, `title`, `text`, and optional assets. Always validate the corpus alongside train rows.

## Mining Workflow Lessons

Tevatron's hard-negative mining utility is reference-only for this skill because it loads full datasets/corpora, may use multiprocessing, and can require retrieval outputs generated by expensive model/index runs. Preserve these behaviors when recreating or auditing a mining workflow:

- It reads ranking rows as `qid docid score` and uses the first `depth` results per query.
- It maps corpus `docid` values to rows and inserts full passage objects into `negative_passages`.
- For relevance-judged data, candidates whose `docid` appears in `positive_passages` are skipped.
- For exact-match QA data, candidates whose text contains an answer span are promoted to positives; candidates without answer matches become hard negatives.
- Output is filtered to keep only queries with at least one positive and at least `min_hn` negatives.
- It concatenates original train rows with mined rows, so callers should understand whether they want augmentation or replacement.

Do not tell a future agent to run the original mining script from the source checkout. If the user needs mining execution, first route first-stage search to `../encoding-retrieval/`, then build a local, task-specific miner using the same input contracts.

## Ranking Inputs for Mining

A first-stage ranking file for conversion or mining should look like:

```text
q1 d2 38.1
q1 d3 12.4
q2 d7 99.0
```

Required consistency checks:

- Every `qid` should exist in the train/query file.
- Every `docid` should exist in the corpus file.
- Known positives should not remain in the final negative list.
- Depth should exceed the target negative count because filtering removes positives, duplicate docids, missing rows, and exact-match answer hits.
- ID types should stay strings; mixed numeric/string IDs cause silent join misses.

Validate local rankings with:

```bash
python scripts/validate_tevatron_jsonl.py --kind ranking --input rank.tsv --corpus corpus.jsonl
```

## Rerank Input Bridge

A reranker input bridge joins retrieval results with query and corpus data and writes one JSON object per query-document pair:

```json
{"query_id":"q1","query":"question text","docid":"d2","title":"optional title","text":"candidate text","score":38.1}
```

Practical requirements:

- Query data must map each `query_id` to a query string. Retriever encoding accepts `query_text`, but reranker inference expects `query`, so normalize the key before scoring.
- Corpus data must map each `docid` to `title` and `text`; use an empty string for missing titles.
- Retrieval depth controls how many candidates per query are emitted.
- Missing query IDs may be skipped by bridge scripts; missing document IDs indicate incompatible corpus/ranking files and should be fixed before scoring.
- Validate the produced file with `python scripts/validate_tevatron_jsonl.py --kind rerank --input rerank.jsonl` before using `../reranking/`.

## Ranking Conversion

Use bundled scripts for local conversion checks:

```bash
python scripts/convert_result_to_trec.py --input rank.tsv --output rank.trec
python scripts/convert_result_to_marco.py --input rank.tsv --output rank.marco.tsv
```

`convert_result_to_trec.py` writes:

```text
qid Q0 docid rank score dense
```

`convert_result_to_marco.py` writes:

```text
qid	docid	rank
```

Both scripts accept three-column Tevatron search rows, four-column `qid docid rank score` rows, and six-column TREC rows. They recompute ranks per contiguous query block and preserve the ordering inside each block.

## Planning Checklist

Before declaring hard-negative data ready, verify:

- The selected train format is explicit: legacy embedded passages or corpus-ID lists.
- `train_group_size` is known and negative depth/count can satisfy it.
- Query IDs match between train/query data and first-stage rankings.
- Corpus IDs match between corpus data, ranking files, qrels, positives, and negatives.
- Qrels use four columns (`qid 0 docid relevance`) when passed to TREC tools.
- Media-bearing negatives have resolvable assets or are intentionally routed to text-only workflows.
