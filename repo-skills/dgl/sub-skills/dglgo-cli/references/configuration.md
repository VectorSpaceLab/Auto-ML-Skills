# DGL-Go Configuration

DGL-Go YAML files are generated by `dgl configure`, `dgl recipe get`, or `dgl configure-apply`. A training config is consumed by `dgl train` and `dgl export`; an apply config is consumed by `dgl apply` and can also be exported.

## Shared Top-Level Keys

- `version`: DGL-Go recipe/config version, commonly `0.0.2` in bundled recipes.
- `pipeline_name`: one of `nodepred`, `nodepred-ns`, `linkpred`, or `graphpred` for training configs.
- `pipeline_mode`: `train` for training/export configs or `apply` for inference configs.
- `device`: Torch device string such as `cpu`, `cuda`, or `cuda:0`.
- `data`: dataset section with at least `name`.
- `general_pipeline`: training or inference runtime settings.

Training configs for `nodepred`, `nodepred-ns`, and `graphpred` use `model`. Training configs for `linkpred` use `node_model`, `edge_model`, and `neg_sampler` instead. Apply configs include `cpt_path` and usually derive the model section from the checkpoint.

## Pipeline-Specific Sections

### `nodepred`

Required sections: `data`, `model`, `general_pipeline`.

Typical model names include `sage`, `gcn`, and `gat`. Typical data names include `cora`, `citeseer`, `pubmed`, and `csv`. The generated `general_pipeline` commonly includes `early_stop`, `num_epochs`, `eval_period`, `optimizer`, `loss`, `save_path`, and `num_runs`.

### `nodepred-ns`

Required sections: `data`, `model`, `general_pipeline`. Neighbor-sampling configs can also include `eval_device`.

`general_pipeline.sampler` should include `name: neighbor`, `fan_out`, `batch_size`, `num_workers`, `eval_batch_size`, and `eval_num_workers`. If `model.num_layers` is set, keep the `fan_out` list length equal to `num_layers`; source generation asserts this before rendering.

### `linkpred`

Required sections: `data`, `node_model`, `edge_model`, `neg_sampler`, `general_pipeline`.

`general_pipeline` commonly includes `hidden_size`, `eval_batch_size`, `train_batch_size`, `num_epochs`, `eval_period`, `optimizer`, `loss`, `save_path`, and `num_runs`. If `data.split_ratio` is set for custom splitting, also set `data.neg_ratio`; the source generator asserts both are present together.

### `graphpred`

Required sections: `data`, `model`, `general_pipeline`.

`general_pipeline` commonly includes `num_runs`, `train_batch_size`, `eval_batch_size`, `num_workers`, `optimizer`, `lr_scheduler`, `loss`, `metric`, `num_epochs`, and `save_path`. Bundled graph prediction recipes often target OGB molecule datasets and may require RDKit and OGB.

## Recipe Catalog

`dgl recipe list` reads bundled YAML files and prints filename, `pipeline_name`, and `data.name`. `dgl recipe get RECIPE.yaml` copies one file into the working directory. Common recipe families include:

- `nodepred_cora_gcn.yaml`, `nodepred_cora_gat.yaml`, `nodepred_cora_sage.yaml`.
- `nodepred_citeseer_*` and `nodepred_pubmed_*` variants.
- `nodepred-ns_arxiv_gcn.yaml` and `nodepred-ns_product_sage.yaml`.
- `linkpred_cora_sage.yaml`, `linkpred_collab_sage.yaml`, and `linkpred_citation2_sage.yaml`.
- `graphpred_hiv_gin.yaml`, `graphpred_hiv_pna.yaml`, and `graphpred_pcba_gin.yaml`.

Recipe configs often use `cuda` or `cuda:0`. Change `device` to `cpu` for CPU-only preflight or small experiments, but expect OGB and molecule recipes to remain dependency- and download-heavy.

## Custom CSV Data

DGL-Go supports `data.name: csv` for custom datasets through DGL CSV loading. The dataset directory should contain a metadata YAML file and CSV files for nodes and edges. For deep CSV layout details, route to `datasets-and-io`.

Training config notes:

- Use `data.data_path` to point to the CSV dataset directory.
- Use `data.split_ratio: [0.8, 0.1, 0.1]` when the dataset has no native train/validation/test masks or splits.
- For link prediction custom splits, set both `data.split_ratio` and `data.neg_ratio`.
- Keep `split_ratio` as three numeric values that sum to approximately `1.0`.

## Checkpoints and Apply Configs

Training templates save checkpoints under `general_pipeline.save_path` as `run_i.pth`, for example `results/run_0.pth`. The checkpoint contains the training configuration and model state dict.

Apply config keys:

- `pipeline_name`: `nodepred`, `nodepred-ns`, or `graphpred`.
- `pipeline_mode`: `apply`.
- `device`: copied from training config by `configure-apply`; edit if needed for the target runtime.
- `data`: apply dataset section. If omitted during `configure-apply`, DGL-Go reuses the training dataset name.
- `cpt_path`: checkpoint path, usually `SAVE_DIR/run_i.pth`.
- `general_pipeline.save_path`: output directory for inference results, defaulting to `apply_results` for node prediction apply configs.

Use `dgl export --cfg apply.yaml --output apply.py` to review or customize the generated inference script before `dgl apply`.
