# Evaluation, Verification, and Handoff

## Purpose

Read this reference after the runtime repo skill draft exists, all sub-skills
have been integrated, and usability test cases have been generated or are about
to be generated by this skill. It defines the content-level self-refine loop,
native repo test/example verification, verification checklist, final coverage
report, publication notes, quality bar, and the artifact-placement rule that
keeps review output out of the runtime skill directory.

## Artifact Boundary

Keep two output areas separate:

- Runtime skill directory: the publishable repo skill, containing only
  `SKILL.md`, `references/`, `scripts/`, `sub-skills/`, and optional runtime
  assets/templates that future agents need to use the skill.
- Review/test artifact directory: all check-only material, defaulting to
  `<repository-path>/skills/tests/<chosen-skill-id>/`.

Within the review/test artifact directory, keep concrete test cases separate
from review and report documents:

```text
<artifact-root>/
  test-cases/              # user_request.txt, README.md, assertions.json, fixtures
    index.md
    sub-skills/<id>/...
    root/...
    integration/...
  reports/
    integration/           # plans and maps from create-repo-skill
    self-refine/           # eval subsets, iterations, assertion grades
    verification/          # native candidates, native runs, static checks
    final/                 # final report, human handoff, publication checklist
```

Write usability test cases only under `test-cases/`. Write self-refine eval
files, native test/example candidate maps, native verification reports, final
coverage reports, automatic verification reports, human-review notes,
publication checklists, prompt samples, benchmark notes, and temporary review
scratch files worth keeping under `reports/`. Do not put `evals/`,
`verification-report.*`, `native-verification-report.*`,
`final-skill-report.*`, `human-review.md`, `publication-checklist.md`,
`prompt-sampling.md`, or similar check artifacts inside the generated skill
directory.

## Evaluate and Self-Refine

After receiving a runtime skill draft, do not stop at first pass. Run a lightweight evaluation loop to catch thin, overpacked, or hard-to-use skills before delivery.

Start with a whole-skill content-level self-refine pass. Review the completed
root skill, every sub-skill, bundled references, bundled scripts, and
repo-provenance together, not as isolated files. Check whether the generated
content satisfies the original user request, the confirmed repository
include/exclude map, the planned sub-skill structure, and each subagent review
rubric. Fix gaps before moving to usability cases or final handoff.

Use the generated usability test case directories as the primary source of realistic user prompts. Read each selected case's `user_request.txt` as the prompt, its `README.md` as the expected-behavior notes, and its `assertions.json` as the grading contract when present. Pick a focused subset when the test set is large, making sure the subset includes:

- At least one novice or broad-purpose prompt.
- At least one expert or specific prompt.
- At least one ordinary workflow.
- At least one troubleshooting or edge-case prompt.

When practical, also save a small machine-readable eval subset under the
review/test artifact directory as `reports/self-refine/evals.json`:

```json
{
  "skill_name": "repo-name",
  "evals": [
    {
      "id": 1,
      "prompt": "Use this repo skill to set up the package and run a minimal inference smoke test.",
      "expected_output": "A correct setup path, verified import command, and smoke-test result.",
      "files": [],
      "assertions": [
        "The output uses the install command documented in the root SKILL.md",
        "The output runs or references the generated smoke-test script",
        "The output reports a concrete import or runtime verification result",
        "The output does not depend on scripts or docs from the original repo checkout",
        "The output does not mention the user's temporary Python environment",
        "All generated skill directory names, frontmatter name fields, and evals skill_name values are lowercase-hyphen identifiers",
        "Every sub-skill frontmatter name exactly matches its sub-skills/<id>/ directory basename, usability target id, and workflow subSkill id when workflow metadata is available",
        "Sub-skill names avoid redundantly repeating the root repo skill id unless needed to disambiguate sibling sub-skills",
        "Every generated root and sub-skill frontmatter description is wrapped in double quotes",
        "Every generated root and sub-skill SKILL.md frontmatter contains disable-model-invocation: true for compatible agents; Codex target visibility is controlled during import",
        "The generated skill tree covers every public capability identified in the coverage matrix",
        "The main-agent sub-skill plan records boundaries, evidence sources, required outputs, cross-links, and acceptance rubrics for each sub-skill",
        "The relevant sub-skill has enough references, scripts, examples, parameters, and troubleshooting to complete the task without reopening the original repo",
        "Copyable commands in references match the bundled scripts' actual option names and file paths",
        "Any high-frequency support workflow such as data preparation, command generation, data-layout checking, or optional dependency diagnosis is reachable from a root route or nearby sub-skill"
      ]
    }
  ]
}
```

For each test prompt, write or load concrete assertions before judging the result. Good assertions check observable outcomes: a file exists, a command is present, an API signature is correct, a workflow references the right sub-skill, a bundled helper is used, a native-evidence-derived behavior is reflected, or troubleshooting advice names the expected failure mode.

When subagents or isolated runs are available, run at least one fresh agent against the draft skill with no access to your research notes. Ask it to complete a test prompt using the generated skill and save or summarize the result. If isolated runs are not available, review from the perspective of a future agent that can only read the generated skill files.

Grade assertions with `PASS` or `FAIL` and cite evidence. Then perform a qualitative review:

- Triggering: would the root and sub-skill `description` fields plausibly trigger for natural prompts from the usability test case `user_request.txt` files, including novice prompts that do not know exact API names?
- Breadth: compare the generated skill tree against the coverage/depth matrix.
- Depth: can a future agent complete common workflows using only the nearest `SKILL.md` plus linked bundled references/scripts?
- Executability: when the skill provides a helper script or template, do the
  documented examples match the helper's actual arguments, expected inputs,
  outputs, and failure codes?
- Workflow fit: are high-frequency support workflows, such as data preparation,
  command generation, data-layout checks, environment checks, optional
  dependency checks, and maintainer guidance, routed at the right level instead
  of being buried in an unrelated sub-skill?
- Routing: can the agent identify the correct sub-skill from the root router?
- Concrete requirements: are commands, imports, config files, data assumptions, and optional dependencies concrete and verified?
- Self-containment: does any workflow force the agent back into the original repo or depend on original repo paths?
- Privacy: does any generated file leak local Python environment details?
- Structure: are identifiers normalized, frontmatter parseable, and bulky material moved into references/scripts?

Save a short evaluation note when practical, for example
`reports/self-refine/iteration-1.md` under the review/test artifact directory,
with prompts, assertions, grades, failures, and revisions made.

Treat `reports/self-refine/` as a development artifact under the review/test
artifact directory, not part of the public runtime skill. Do not link
self-refine artifacts from generated root or sub-skill `SKILL.md` files as
user-facing documentation.

Treat usability test case directories under `test-cases/` in the configured
review/test artifact directory as review artifacts. They may be kept and shared
as a test suite when useful, but they are not required for the generated skill
to work at runtime.

If any assertion fails or the qualitative review reveals a gap, revise the generated skill before finishing and repeat the focused check.

## Native Repo Test/Example Verification

After content-level self-refine and before final static verification, use the
original repository's native examples/tests as ground-truth verification when
safe. This happens only after all sub-skills have been generated, reviewed, and
integrated into a complete runtime skill. Do not run native examples/tests as a
sub-skill drafting shortcut.

Start from the native test/example candidate map produced during repository
analysis or whole-skill integration. If no map was provided, build a small one
from available repo evidence before running anything. Candidate sources include
repo-owned examples, tests, CLI help snippets, notebooks, smoke scripts,
fixtures, golden files, and documented commands.

Select a representative safe subset:

- Prefer short tests, CLI `--help`/parser checks, import checks, tiny fixtures,
  smoke scripts, and examples with documented minimal inputs.
- Include at least one native candidate for each major workflow when a safe
  candidate exists.
- Include support-workflow candidates such as validation, conversion,
  data-layout, config, and optional-dependency checks when those workflows are
  important.
- Skip candidates that require network access, credentials, private data, large
  downloads, unavailable hardware, long training, benchmarks, destructive
  writes, or unknown side effects unless the user explicitly approves.

Record candidate and execution results under
`<artifact-root>/reports/verification/`, for example:

```text
reports/verification/native-ground-truth-candidates.md
reports/verification/native-ground-truth-candidates.json
reports/verification/native-verification-report.md
reports/verification/native-verification-report.json
```

Use these result statuses:

- `PASS`: the native case ran successfully and the generated skill covers the
  corresponding capability with matching guidance.
- `SKILL_GAP`: the native case ran successfully, but the generated skill lacks
  coverage, routes to the wrong area, uses stale flags or APIs, omits necessary
  prerequisites, or fails to provide enough depth.
- `NATIVE_FAIL`: the native case failed in the current environment. Record the
  command, relative repo evidence path, stdout/stderr summary, exit code, and
  whether the failure implies a skill issue, an environment issue, or a repo
  issue.
- `SKIP_UNSAFE`: the candidate was intentionally not run for safety,
  credentials, hardware, network, data, cost, or runtime reasons. Record the
  reason and decide whether a synthetic assertion-backed usability case covers
  the same capability.
- `SKIP_NOT_SELECTED`: the candidate was not part of the representative subset.
  Record enough rationale to show that major workflows still received coverage.

For every `SKILL_GAP`, revise the runtime skill and rerun the affected check or
replace it with an equivalent focused review when rerun is impractical. For
every `NATIVE_FAIL`, do not mark the skill as verified until you decide whether
the failure contradicts a generated skill claim. If native execution is skipped,
do not count it as pass; report the skip and the synthetic coverage used
instead.

When using a helper runner, keep it generic and manifest-driven. A runner may
execute a preselected safe manifest with timeouts and write JSON output, but it
must not guess that arbitrary repo tests are safe. Store local absolute command
paths only in private artifacts when needed; public runtime skill files must
not mention local checkout paths or inspection environment paths.

## Static Verification and Human Review

After the self-refine pass, run a static verification pass over the generated
skill tree and review/test artifacts. DisCo does not require an external
Python verifier for this workflow; perform the checks below directly and save a
short review package under `<artifact-root>/reports/` when practical.

Recommended review artifacts:

- `reports/verification/verification-report.json` or
  `reports/verification/verification-report.md`:
  PASS/WARN/FAIL checks
  with evidence.
- `reports/verification/native-verification-report.json` or
  `reports/verification/native-verification-report.md`: selected original repo
  examples/tests, statuses, skip reasons, and skill coverage decisions.
- `reports/final/final-skill-report.md`: final comparison of original repo
  capabilities, generated skill coverage, usability assertions, native
  verification results, and long-tail gaps.
- `reports/final/human-review.md`: concise review summary for a human reviewer.
- `reports/final/publication-checklist.md`: pre-publication checklist.
- `reports/final/prompt-sampling.md`: quick samples from generated usability prompts.

Treat this review package as a quality gate. Fix critical or high failures
before finishing unless the user explicitly accepts the risk.

The static pass checks frontmatter validity, canonical IDs, double-quoted
descriptions, disabled model invocation for generated repo root/sub-skills,
router model visibility, root and sub-skill routing, relative links,
reference/script reachability, install/import guidance, local path leaks,
source-repo runtime dependencies, usability test-case shape, copyable prompts,
README review sections, optional eval shape, generated cache/build debris,
script/example argument consistency, and review-package readiness.

## Verification Checklist

Before finishing, confirm:

- The directory layout matches the intended output structure.
- Each generated `SKILL.md` has valid frontmatter.
- Every frontmatter `description` is double-quoted, and YAML-sensitive strings are escaped.
- Every generated repo root and sub-skill `SKILL.md` contains
  `disable-model-invocation: true`.
- The live or generated `repo-skills-router/SKILL.md` does not contain
  `disable-model-invocation: true`, because it must remain the routing entry
  point for compatible agents.
- The generated root directory basename is a canonical lowercase-hyphen skill id, not a raw mixed-case repo name.
- Every `name` field uses only lowercase letters, numbers, and hyphens, is no longer than 64 characters, and matches the corresponding generated skill directory basename.
- Every sub-skill directory basename under `sub-skills/`, sub-skill
  frontmatter `name`, coverage/depth matrix output location, subagent review
  note, workflow `subSkill` id when available, and usability case target id use
  the same canonical sub-skill id.
- Sub-skill ids do not redundantly repeat the generated root skill id or repo
  name unless the plan records why that repetition is needed to distinguish two
  sibling routes.
- `reports/self-refine/evals.json` under the review/test artifact directory uses the
  canonical root skill id in `skill_name` when that file exists.
- Root `SKILL.md` routes to sub-skills instead of duplicating all details when sub-skills exist.
- Every sub-skill `SKILL.md` routes to nearby references and scripts instead of becoming a manual.
- Root `SKILL.md` includes package installation instructions and a minimal verification command.
- Root `SKILL.md` links to `references/repo-provenance.md` for freshness checks.
- `references/repo-provenance.md` exists, uses schema `disco.repo-provenance.v1`, and records source commit or an explicit non-Git source identifier, dirty state, package version when available, and relative evidence paths.
- Package installation instructions are public and reproducible, not copied from the user's temporary Python environment.
- The generated skill covers every user-facing capability found in the repo, docs, examples, tests, package metadata, and installed package inspection.
- The coverage/depth matrix has no unmapped public capabilities.
- The sub-skill plan is strong enough to explain why each sub-skill exists,
  what it owns, what it excludes, which evidence supports it, which
  references/scripts it should contain, and how the main agent accepted it.
- Whole-skill integration artifacts exist when generated by the calling
  workflow, including integration notes, coverage/depth matrix, native
  candidate map, and long-tail gap register.
- Every non-trivial public capability maps to a root or sub-skill entry plus enough bundled references/scripts for practical use.
- Each sub-skill passes a depth check: workflows, APIs, commands, configs, inputs/outputs, examples, troubleshooting, and safe scripts are sufficient for future agents to avoid reopening original source code or docs for ordinary use.
- Every high-frequency support workflow identified in the evidence map, such
  as data preparation, schema validation, conversion, command generation,
  data-layout checking, optional dependency checking, environment checking, or
  maintainer policy, is either routed as its own sub-skill or linked from the
  nearest owning sub-skill with enough depth to be usable.
- Sub-skill boundaries are explained, functionally independent, and not just copied from folder names.
- References and scripts are linked from a nearby `SKILL.md` with clear read/run guidance.
- Every linked reference or script path resolves inside the generated skill directory.
- Every local Markdown link in runtime skill files resolves inside the generated
  skill directory and points to an existing bundled runtime file.
- Every source repo script, example, notebook, tool, or config that a future
  agent is told to run, read, or adapt has a bundled replacement in the skill's
  own `scripts/` or `references/` tree.
- Runtime references distinguish source repo artifact names from bundled helper
  names when they differ, so future agents do not try to run nonexistent source
  paths.
- Copyable examples that invoke bundled scripts or templates use option names,
  positional arguments, input files, output files, and config keys accepted by
  those bundled assets.
- Bundled diagnostic scripts can run from an arbitrary current working
  directory or document the required `--repo-root`/package-install assumption,
  and missing optional dependencies are reported as actionable diagnostics
  rather than unexplained tracebacks when practical.
- Generated `SKILL.md` files do not instruct future agents to run, read, or open scripts, docs, examples, notebooks, or configs from the original repo checkout.
- Generated skill files do not mention the user's local Python executable, virtualenv or conda name, activation command, machine-specific path, local checkout path, or `pip show` installation directory.
- `references/repo-provenance.md` does not mention local absolute paths, conda prefixes, Python executables, private remote URLs, cache paths, credentials, or `pip show Location` values.
- `references/` is non-empty wherever repo docs, examples, notebooks, tests, or CLIs provided reusable material.
- `scripts/` exists wherever the repo has reusable scripts, example pipelines, repeatable diagnostics, or smoke-test opportunities that can be safely adapted.
- `troubleshooting.md` exists in relevant `references/` directories when troubleshooting information is available, or the user was asked when the repo alone was insufficient.
- No `SKILL.md` exceeds its intended size without a clear reason.
- No detailed content is duplicated between `SKILL.md` and reference files.
- Common workflows can be completed from the generated skill tree and a freshly installed public package environment without rereading the original repo.
- Usability test case directories were generated under
  `<artifact-root>/test-cases/`.
- Usability test cases include diverse user familiarity levels, diverse scenario difficulty, and broad coverage of major sub-skills and public capabilities.
- Every generated sub-skill has one or two new difficult synthetic cases under
  `test-cases/sub-skills/<sub-skill-id>/`, in addition to any cases directly
  derived from original repo tests/examples.
- The complete integrated skill has one or two difficult integration cases
  under `test-cases/integration/`. These prefer original repo tests/examples as
  source scenarios; if synthesized, the case README and assertions explain why
  no suitable native integrated candidate existed.
- Every usability test case is a directory named with the target root or sub-skill id plus a descriptive scenario slug.
- Every usability test case directory contains `user_request.txt` with only the directly copyable future-user prompt.
- Every usability test case directory contains `README.md` with user persona, scenario coverage, expected successful behavior, and failure signals.
- Every usability test case directory contains `assertions.json`, unless a
  specific case is documented as prompt-sampling only and excluded from quality
  gates.
- Assertions are concrete, evidence-backed when possible, and gradeable as
  `PASS` or `FAIL`.
- Weak generic prompts were avoided or rewritten into scenario prompts with
  concrete constraints, files, errors, APIs, CLI flags, configs, or expected
  outputs.
- Multiple scenario directories exist for the same sub-skill when that sub-skill has multiple important scenarios.
- The generated root and sub-skill `description` values are broad enough to trigger on the natural prompts represented by the usability test cases.
- `test-cases/index.md` or the equivalent user-specified index summarizes case coverage, difficult synthetic coverage per sub-skill, integrated difficult cases, original repo-native anchors, and untested major capabilities.
- Usability test cases are not linked from generated `SKILL.md` files as runtime documentation unless the user explicitly requested that.
- The eval/self-refine pass was performed, assertions were graded with evidence, and gaps were fixed or explicitly reported.
- Native repo examples/tests were considered after whole-skill integration.
- Safe selected native candidates were run, or skipped with explicit safety,
  hardware, data, credential, network, runtime, or scope reasons.
- Native verification results were compared against generated skill coverage,
  not treated as independent environment checks only.
- `SKILL_GAP` native results were fixed or explicitly accepted by the user.
- Native skips were not counted as passes and were paired with synthetic
  assertion-backed coverage when needed.
- The automatic verifier was run when available, the human review package was created, and critical/high failures were fixed or explicitly accepted.
- `reports/self-refine/`, `reports/verification/`, `reports/final/`,
  verification reports, human-review notes, publication checklists, prompt
  samples, staleness audits, and benchmark notes are under the review/test
  artifact directory, not inside the generated runtime skill.
- The artifact root is clean: concrete cases are under `test-cases/`, reports
  and review documents are under `reports/`, and no loose final-deliverable
  files or case directories are mixed directly into the artifact root.
- The generated runtime skill directory contains no check-only directories or
  files such as `evals/`, `tests/`, `review/`, `verification-report.*`,
  `human-review.md`, `publication-checklist.md`, or `prompt-sampling.md`.
- The generated runtime skill directory contains no Python cache files,
  `__pycache__/` directories, `*.pyc`, `*.pyo`, build outputs, downloaded
  caches, or other generated debris that is not required runtime skill content.
- The final handoff includes a short verification report covering repo evidence used, generated sub-skills, usability test case coverage, native verification results, final coverage report path, verification results, and self-refine revisions.
- Safe import/signature checks were run in the provided Python environment for APIs mentioned in the skill, then translated into public package facts.
- Stale placeholders, speculative claims, and unnecessary empty directories were removed.
- A repo skill routing update is ready for `repo-skills-router` if import is approved or auto-authorized, including usage-scenario placement plus in-scenario repo role, difference, and selection guidance when relevant.

## Final Handoff

When presenting the verified skill to the user, include a short verification report. Keep it concise, but do not omit these items:

- Repo evidence used: summarize the repository state and evidence categories consulted, such as source commit or non-Git source identifier, dirty state, source roots, docs, examples, scripts, tests, configs, existing skills, and installed-package inspection.
- Generated skill content: give the generated skill path and summarize the root skill plus each generated sub-skill, including its purpose and main covered capabilities.
- Review/test artifacts: give the artifact root path, the `test-cases/` path,
  the `reports/` path, the usability case count, the per-sub-skill or
  per-capability distribution, the one or two difficult synthetic cases per
  sub-skill, the one or two integrated difficult cases, representative scenario
  names, native verification report path, final skill report path, and the
  report/eval artifact paths created.
- Self-refine result: summarize the lightweight checks performed, important revisions made, and any explicit remaining gaps.
- Native verification result: summarize selected original repo examples/tests,
  PASS/SKILL_GAP/NATIVE_FAIL/SKIP counts, and any skill changes made because of
  native evidence.

Also include a short publication note:

- The generated `SKILL.md`, `sub-skills/`, `references/`, `scripts/`, and
  optional runtime assets/templates are the public skill content.
- The generated usability test case directories under `test-cases/` and the
  report package under `reports/` in the configured review/test artifact
  directory are review cases and quality-gate evidence for checking how future
  users may use the skill.
- Because check artifacts are outside the runtime skill directory by default,
  publishing or importing the skill directory should not include self-refine
  notes, verification reports, human-review notes, publication checklists, or
  prompt samples unless the user intentionally publishes the artifact suite.

## Import and Repo Skills Router

After the generated repo skill passes verification, resolve the import decision
policy from the calling workflow. If the policy is missing, default to
`importAfterVerification: ask`.

When `importAfterVerification` is `ask`, use `ask_user_question` when available
to ask whether to import it into DisCo's user skills directory at
`~/.disco/agent/skills/`. Present the verified runtime skill directory, the
review/test artifact directory, and a concise verification summary so the user
can make an informed decision. Do not only ask in a normal assistant message and
stop when the structured question tool is available.

When `importAfterVerification` is `auto-import`, do not ask again after a
successful verification. Auto-import only when the skill is import-ready and no
high or critical verification failure remains unresolved. In the handoff and
final report, state that the original create request authorized auto-import.

If the user approves import or import is auto-authorized, run the import as one
global locked transaction using `scripts/with_import_lock.mjs`. The lock path is
under the DisCo agent directory, resolved from `DISCO_CODING_AGENT_DIR`
when set, otherwise `~/.disco/agent/locks/repo-skills-import.lockdir`. The
lock must be shared by all create, refresh, and verification sessions that write
the managed skills library. Hold it across the complete mutation of
`<agent-dir>/skills/`, not only the final router write.

While holding the lock:

1. Resolve `<agent-dir>` again from `DISCO_CODING_AGENT_DIR` or
   `~/.disco/agent`, then set `<skills-root>` to `<agent-dir>/skills`.
2. Re-read the current `<skills-root>/<skill-id>/` and live
   `<skills-root>/repo-skills-router/` state. Do not rely on a router snapshot
   prepared before the lock was acquired; another session may have imported a
   skill while this verification run was waiting.
3. Re-check overwrite safety for `<skills-root>/<skill-id>/`. If it already
   exists and this workflow is not explicitly updating that exact skill, ask
   before replacing it or choose a non-conflicting import name when that is
   consistent with the generated skill id policy.
4. Copy only the runtime skill directory into
   `<skills-root>/<skill-id>/`. Do not copy usability cases, evals, review
   reports, publication notes, prompt samples, or other check-only artifacts
   into the user skills directory.
5. Ensure `<skills-root>/<skill-id>/references/repo-routing-metadata.json`
   exists. This JSON is the source of truth for the imported skill's router
   scenario placement, role, read-when signals, best-fit tasks, avoid-when notes,
   useful entry points, and selection guidance.
6. Check that the routing metadata uses a canonical scenario ID or alias from
   `<skills-root>/repo-skills-router/references/scenario-registry.json` when the
   registry exists. If the metadata introduces a new scenario, it must include a
   top-level `scenarios.<id>` entry with `allow_new: true`,
   `why_not_existing`, and `expected_future_reuse`; otherwise the updater should
   reject the import.
   Also lint metadata fields before import: field values must not include
   rendered labels such as `Read when`, `Avoid when`, `Best for`, or `Role`, role
   text must not start with numeric artifacts, and `useful_entry_points` must not
   contain generated count placeholders such as `1 more sub-skills`.
7. Run `scripts/update_repo_skills_router.mjs --agent-dir <agent-dir> --already-locked`
   inside the same locked process. Do not hand-edit the router Markdown as the
   import mechanism. The updater re-reads live skills and the live router,
   creates the router from the bundled template when missing, rebuilds the root
   quick map, `references/usage-scenarios.md`,
   `references/maintenance.md`, `references/scenario-registry.json`, and every
   `references/scenarios/<scenario>.md` page, removes legacy side-channel router
   files, normalizes aliases through the scenario registry, and validates
   coverage and links before returning success.
8. Treat a non-zero updater exit as import failure before releasing success.
   Before reporting success, verify that the imported skill directory exists and
   that the updater output says the router was updated.

Do not treat `auto-import` as permission to overwrite an unrelated existing
managed skill. If `~/.disco/agent/skills/<skill-id>/` already exists and
this workflow is not explicitly updating that exact skill, ask before replacing
it or choose a non-conflicting import name when that is consistent with the
generated skill id policy.

Do not search for a same-named `repo-skills-router` directory in another agent
skills root before the DisCo user copy has been created or updated. The
DisCo user copy is authoritative for the primary import. If it does not
exist yet, create it from the bundled `repo-skills-router` template, then apply
the routing update.

Prepare the `repo-skills-router` update as structured progressive routing
metadata for future agent routing. It should name the repo or project, identify
practical usage scenarios, place the skill on one or more canonical scenario
pages, and explain how to choose among overlapping imported repo skills inside
those scenario pages. Store that metadata in
`references/repo-routing-metadata.json` in the runtime skill before import. A
minimal metadata file can look like:

```json
{
  "skills": {
    "flag-embedding": {
      "scenarios": [
        {
          "id": "embedding-retrieval-workflows",
          "title": "Embedding Retrieval Workflows",
          "when_to_read": "Embedding, reranking, vector search, retrieval training, and corpus indexing tasks.",
          "role": "Guides FlagEmbedding and BGE embedding, reranking, and retrieval workflows.",
          "read_when": "The request names FlagEmbedding or BGE, or asks for embedding, reranking, retrieval training, vector search, or BGE-style model workflows.",
          "best_for": "Embedding model setup, inference, fine-tuning, reranking, and retrieval evaluation.",
          "avoid_when": "The request is about a different retrieval framework or a non-embedding model family.",
          "useful_entry_points": ["flag-embedding/SKILL.md"],
          "selection_guidance": "Choose `flag-embedding` for BGE/FlagEmbedding setup, inference, fine-tuning, reranking, retrieval evaluation, and related API or config troubleshooting."
        }
      ]
    }
  }
}
```

Metadata must route from task evidence, not only package-name mentions. Package
names, repo names, and imports are strong signals, but `read_when` and
`selection_guidance` must also include the workflows, data/model formats,
API/CLI surfaces, configs, artifacts, and error modes that imply this repo skill
when the user describes a need without naming the package. Do not write
ambiguous `Choose this skill` or `Choose it` wording; name the concrete skill id
or package. Metadata values are not rendered Markdown rows: do not start them
with labels like `Read when`, `Avoid when`, `Best for`, or `Role`; do not start
`role` with numeric artifacts such as `0 for` or `1 workflows`; and do not use
non-path `useful_entry_points` placeholders such as `1 more sub-skills`.

Use these checks before approving import or publication:

```bash
rg -n '"role"\s*:\s*"(Role\b|[0-9]+\b)|"read_when"\s*:\s*"Read when\b|"avoid_when"\s*:\s*"Avoid when\b|"best_for"\s*:\s*"Best for\b|"useful_entry_points"\s*:\s*\[[^\]]*"[0-9]+ more sub-skills"' \
  <skills-root>/*/references/repo-routing-metadata.json
```

After rebuilding the router, also check rendered Markdown:

```bash
rg -n '^(Role|Read when|Best for|Avoid when|Useful entry points):\s*(Role|Read when|Best for|Avoid when|Useful entry points)\b|^Role:\s*[0-9]+\b' \
  <skills-root>/repo-skills-router
```

When import is approved or auto-authorized, update the DisCo user
`repo-skills-router` skill only inside the global import lock and only through
`scripts/update_repo_skills_router.mjs`. The generated router keeps its root quick
map compact, updates `references/usage-scenarios.md`, and updates each relevant
`references/scenarios/<scenario>.md` page with the repo skill's role, read-when
signals, best-fit tasks, avoid-when notes, useful entry points, and selection
guideline. It also preserves `references/scenario-registry.json`, normalizes
alias scenario IDs to canonical pages, and rejects unknown scenarios unless the
metadata includes explicit new-scenario rationale. If the imported skill
overlaps with existing repo skills, put the comparison in the same scenario page
under `How To Choose` through the structured metadata. Do not create or depend
on a separate similar-skill comparison file.

Do not automatically synchronize the runtime skill or router into other agent
tools as part of this verification/import workflow. If the user explicitly asks
to export DisCo's managed skill library to another agent such as
`~/.codex`, `~/.agents`, or `~/.claude`, use the sibling
`import-repo-skills-to-agent` meta skill. That skill handles target skills-root
detection, duplicate-skill overwrite questions, and merging an existing target
`repo-skills-router` with the DisCo-managed router.

## Quality Bar

The generated skill must be:

- Consistent: aligned with the original repo's implementation, public APIs, terminology, and documented behavior.
- Correct: free of false API claims, invalid commands, broken imports, inaccurate signatures, or fabricated capabilities.
- Comprehensive: makes all user-facing capabilities reachable through routers, sub-skills, references, and scripts.
- Deep enough: each non-trivial sub-skill contains or links to enough bundled guidance for future agents to perform common tasks without returning to the original repo.
- Self-contained: does not depend on source repo docs, examples, notebooks, configs, or scripts remaining available after skill generation.
- Public-ready: omits user-specific inspection environment details and describes only reproducible public installation and runtime requirements.
- Well-structured: keeps `SKILL.md` files readable and moves depth into the nearest `references/` or `scripts/` directory.
- Tested: includes usability test case directories plus a lightweight eval/self-refine pass.
- Reasonable: free of vague, contradictory, misleading, or unjustified guidance that could confuse future agents.
